# 媒体库歌手页设计规格

## 背景

当前播放页只展示歌曲名和歌手名，用户想从播放页点击歌手进入一个更完整的歌手页，而不是依赖歌曲详情弹窗。歌曲详情弹窗空间较小，不适合作为歌手内容的主要承载入口。

第一版目标是提供一个轻量、稳定、可继续扩展的媒体库歌手页：从播放页进入，只展示 App 内媒体库中该歌手的歌曲，并支持直接播放。

## 目标

- 在播放页点击歌手名进入歌手页。
- 歌手页查询范围限定为 App 内媒体库歌曲。
- 歌手页第一版展示歌手名、歌曲数量、播放全部按钮和歌曲列表。
- 支持联名歌手拆分选择，避免只能按完整歌手字段查找。
- 支持点击歌曲播放，并将当前歌手页歌曲列表作为播放队列。
- 在线歌曲也允许点击歌手，但结果仍只来自 App 内媒体库。

## 非目标

- 不做全局在线音源歌手中心。
- 不跳转第三方在线源歌手页或外链页面。
- 不在第一版加入专辑 Tab、来源 Tab、统计画像或播放历史分析。
- 不用歌曲详情弹窗作为第一版主入口。
- 不在第一版合并或去重跨来源重复歌曲。

## 入口设计

### 播放页入口

竖屏播放页和横屏播放页的 Header 中，歌手名文本都改为可点击区域：

- `src/screens/PlayDetail/Vertical/components/Header.tsx`
- `src/screens/PlayDetail/Horizontal/components/Header.tsx`

点击行为保持一致：

1. 读取当前播放歌曲的 `singer` 字段。
2. 如果 singer 为空或标准化后为空，提示用户当前歌曲没有可查看的歌手信息，不进入歌手页。
3. 如果 singer 可拆分为单个歌手，直接按该歌手查询媒体库。
4. 如果 singer 可拆分为多个歌手，弹出底部选择面板，让用户选择要查看的歌手或完整匹配项。
5. 查询媒体库后如果命中歌曲，进入歌手页；如果没有命中，toast 提示媒体库中未找到该歌手歌曲，不进入空页。

### 在线歌曲入口规则

当前播放歌曲即使来自在线源，也允许点击歌手名。该入口只使用在线歌曲的歌手字段作为查询条件，查询范围仍限定在 App 内媒体库：

- 媒体库中存在该歌手歌曲：进入媒体库歌手页。
- 媒体库中不存在该歌手歌曲：toast 提示，不跳转在线源，不进入空页。

## 联名歌手拆分规则

播放页歌手字段需要支持常见联名写法拆分。第一版使用确定性字符串规则，不引入网络查询或复杂艺名词典。

### 分隔符

默认识别以下分隔符：

- `、`
- `/`
- `;`
- `；`
- `,`
- `，`
- `&`
- `feat.`
- `ft.`
- `with`

英文分隔词按大小写不敏感处理。

### 标准化

拆分后对候选歌手执行：

- 去掉首尾空白。
- 丢弃空字符串。
- 按标准化后的文本去重。
- 如果最终只剩一个候选歌手，直接进入该歌手查询流程。
- 如果最终存在多个候选歌手，显示底部选择面板。

### 底部选择面板

当歌手字段拆分出多个候选歌手时，展示底部选择面板：

- 标题：`查看哪个歌手？`
- 选项：拆分后的每个歌手。
- 额外选项：原始 singer 文本的完整匹配项，例如 `周杰伦 / 方文山（完整匹配）`。

选择拆分歌手时，按 token 精确匹配查询；选择完整匹配项时，只匹配完整 singer 字段。

## 匹配规则

### token 精确匹配

用户选择单个歌手 token 时，媒体库查询采用“拆分后精确匹配歌手 token”：

- 查询 `周杰伦` 时，命中 `周杰伦`。
- 查询 `周杰伦` 时，命中 `周杰伦 / 方文山`。
- 查询 `周杰伦` 时，命中 `周杰伦 feat. 林俊杰`。
- 查询 `周杰伦` 时，不命中 `周杰`。
- 查询 `周杰伦` 时，不命中 `周杰伦乐队`。

该规则的意图是支持联名歌曲，同时避免包含匹配带来的误命中。

### 完整匹配

用户选择完整匹配项时，只命中 singer 字段标准化后与原始 singer 文本标准化后完全相等的歌曲。例如选择 `周杰伦 / 方文山（完整匹配）` 时，不命中只包含 `周杰伦` 的歌曲。

## 媒体库查询范围

歌手页查询范围是 App 内媒体库歌曲，包括但不限于：

- 本地媒体库歌曲。
- WebDAV 歌曲。
- SMB 歌曲。
- OneDrive 歌曲。
- 其他已纳入 App 媒体库管理的歌曲。

第一版不把“当前播放队列”或“当前 UI 列表”作为查询边界。查询应从当前 App 已持有的媒体库数据来源取数，保证从播放页进入时能看到完整媒体库范围内的同歌手歌曲。

## 歌手页页面结构

第一版采用轻量独立页面，不使用歌曲详情弹窗承载。

页面内容：

1. 返回按钮。
2. 歌手名标题。
3. 歌曲数量，例如 `共 12 首歌曲`。
4. `播放全部`按钮。
5. 歌曲列表。

歌曲列表沿用项目现有音乐列表展示组件或视觉风格，避免为第一版新增复杂视觉体系。列表项至少需要展示歌曲名和歌手名；如果现有列表项已经展示专辑、来源或时长，可以按现有组件能力自然保留，但这些字段不是第一版核心要求。

## 播放行为

### 点击单曲

在歌手页点击第 N 首歌曲时：

1. 将当前歌手页命中的完整歌曲列表设置为播放队列。
2. 从第 N 首开始播放。
3. 播放页、迷你播放器和队列状态按现有播放系统规则更新。

### 播放全部

点击 `播放全部` 时：

1. 将当前歌手页命中的完整歌曲列表设置为播放队列。
2. 从第一首开始播放。

### 排序规则

歌手页歌曲排序沿用媒体库当前列表顺序。第一版不按歌名、专辑、曲目号或播放热度重新排序，避免打乱用户已有媒体库顺序，也避免引入字段缺失导致的不稳定排序。

### 重复歌曲规则

第一版保留所有命中的媒体库歌曲，不做去重。若同一首歌曲同时存在于本地、WebDAV、SMB 或其他来源中，歌手页全部展示。这样可以忠实反映媒体库真实数据，避免误合并不同来源、不同音质或不同版本。

## 空状态和错误处理

### 入口前无命中

点击播放页歌手后，如果媒体库查询结果为空，不进入歌手页，直接 toast 提示。建议文案：

- `媒体库中未找到该歌手的歌曲`

### 无歌手信息

如果当前播放歌曲没有 singer 信息，toast 提示并保持在播放页。建议文案：

- `当前歌曲没有歌手信息`

### 查询失败

如果媒体库查询发生异常，toast 提示并保持在播放页。建议文案：

- `歌手歌曲加载失败`

异常应记录到现有日志系统或调试日志中，便于定位媒体库数据问题。

## 路由与状态

新增歌手页应作为独立页面接入现有导航体系。进入页面前完成一次媒体库命中检查，避免展示空页面。

推荐路由参数包含：

- `artistName`：用于页面标题展示的歌手名。
- `matchMode`：`token` 或 `exact`，表示 token 精确匹配或完整字段匹配。
- `sourceSinger`：原始 singer 文本，用于完整匹配和调试。

页面进入后可以重新读取媒体库并计算列表，或由入口传入已命中的歌曲列表。实现时优先遵循项目现有导航与播放队列模式。如果现有导航不适合传递大型歌曲对象，应传递查询参数并在页面内重新查询，避免路由状态过重。

## 组件与职责边界

建议拆分为以下职责单元：

1. 歌手字符串工具
   - 负责 singer 标准化、联名拆分、候选歌手去重、token 匹配和完整匹配判断。
   - 该工具应是纯函数，便于单元测试。

2. 媒体库歌手查询方法
   - 负责从 App 内媒体库歌曲集合中筛选目标歌手歌曲。
   - 输入为歌曲集合、目标歌手、匹配模式和原始 singer。
   - 输出为按媒体库当前顺序排列的歌曲数组。

3. 播放页入口组件改造
   - 负责把歌手文本包装为可点击区域。
   - 负责触发单歌手查询、底部选择面板和 toast。
   - 不直接承载歌手页 UI。

4. 歌手页页面组件
   - 负责展示标题、歌曲数量、播放全部和歌曲列表。
   - 负责调用现有播放队列能力。
   - 不负责解析 singer 字符串的底层规则。

这种边界可以避免把联名拆分、媒体库筛选、页面展示和播放队列操作混在同一个大组件里。

## 国际化

新增用户可见文案需要接入现有 i18n 体系。第一版至少新增或复用以下文案键：

- `artist_page_title`
- `artist_page_song_count`
- `artist_page_play_all`
- `artist_page_choose_artist_title`
- `artist_page_exact_match_option`
- `artist_page_empty_in_library`
- `artist_page_no_artist_info`
- `artist_page_load_failed`

具体 key 命名可按项目现有命名规范调整，但同一含义不要散落多个重复 key。

## 测试要求

### 纯函数测试

覆盖 singer 拆分和匹配规则：

- 单歌手不弹选择。
- `周杰伦 / 方文山` 拆成 `周杰伦` 和 `方文山`。
- `周杰伦 feat. 林俊杰` 支持大小写不敏感拆分。
- 空白、重复和空 token 会被清理。
- token 精确匹配能命中联名歌曲。
- token 精确匹配不会误命中包含关系，例如 `周杰伦` 不命中 `周杰伦乐队`。
- 完整匹配只命中完整 singer 字段。

### 查询测试

覆盖媒体库歌手查询：

- 返回所有命中歌曲。
- 保留媒体库输入顺序。
- 保留重复来源歌曲。
- 空结果返回空数组。

### 页面和入口测试

根据项目现有测试能力覆盖：

- 播放页点击单歌手时进入歌手页。
- 播放页点击多歌手时出现底部选择面板。
- 媒体库无命中时显示 toast 且不进入歌手页。
- 歌手页点击单曲时以当前歌手页列表作为播放队列并从该歌曲开始播放。
- 播放全部从第一首开始播放。

如果当前项目对 React Native 页面测试覆盖有限，第一阶段应至少保证工具函数和查询方法有稳定自动化测试，入口与播放行为可通过手工验证补齐。

## 验收标准

- 竖屏播放页和横屏播放页均可点击歌手名。
- 单歌手可直接进入独立歌手页。
- 多歌手会弹出底部选择面板，并包含完整匹配选项。
- 歌手页只展示媒体库范围内的歌曲，不跳在线源。
- 媒体库无命中时 toast 提示，不进入空页。
- 歌手页展示返回、歌手名、歌曲数量、播放全部和歌曲列表。
- 点击歌曲会以歌手页歌曲列表作为播放队列，从所点歌曲开始播放。
- 点击播放全部会以歌手页歌曲列表作为播放队列，从第一首开始播放。
- 歌曲排序保持媒体库当前列表顺序。
- 重复来源歌曲全部保留。
- singer 拆分、匹配和查询逻辑有自动化测试覆盖。

## 后续扩展方向

第一版完成后，可以在不破坏现有入口和查询规则的前提下扩展：

- 专辑 Tab：按 album 聚合该歌手歌曲。
- 来源 Tab：按本地、WebDAV、SMB、OneDrive 等来源聚合。
- 统计 Tab：展示播放次数、最近播放、年度趋势等画像。
- 歌手图片或封面墙。
- 当前歌手页内搜索和排序切换。
