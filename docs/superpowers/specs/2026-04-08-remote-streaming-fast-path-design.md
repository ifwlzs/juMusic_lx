# 远端流式同步首批可见与播放快路径设计

日期：2026-04-08
状态：已确认，待写实现计划

## 概述

`OneDrive / WebDAV / SMB` 的二期远端流式同步框架已经落地，但当前主线实现仍有一个明显体验落差：

1. 同步主流程仍偏向“先把整轮候选列表枚举完成，再开始逐首补全”；
2. `hydrateCandidate()` 很容易落到“下载临时文件再读元数据”的重路径；
3. 导致用户在大目录或 `OneDrive` 这种高延迟来源上，仍然要等待较久才能看到第一批歌曲进入歌单；
4. 播放侧虽然已经有“当前曲缓存 + 下一首预读”，但与同步侧还没有真正围绕“首批尽快可见、当前播放绝对优先”做优先级收敛。

本次不是重做二期，不新增新 provider，也不引入代理、转码、离线下载中心，而是在现有 `Remote Streaming Sync` 框架上补一层真正的“快路径”：

1. 枚举阶段改成流式产出，而不是先攒完整数组；
2. 单曲补全优先走轻字段和轻量元数据，不够时才走重读取；
3. 只要歌曲满足 `标题 + 歌手 + 时长`，就尽快进入批量提交器并出现在歌单中；
4. UI 层保持当前结构，只补足用户能感知到的渐进可见、后台通知和同步状态反馈。

## 目标

1. 让远端大目录导入时，第一批歌曲无需等待整轮枚举结束就能进入歌单。
2. 保持现有二期统一框架，不为 `OneDrive` 单独开一套同步模型。
3. 将 `hydrateCandidate()` 的默认路径前移到轻量补全，重读取只保留为兜底。
4. 保持“旧歌单先保留、新歌逐步追加、删除最后统一处理”的现有用户规则。
5. 保持“当前曲缓存 + 下一首预读”，但进一步明确任务优先级，避免预读反拖慢首批歌曲出现。
6. 让用户在媒体来源管理器、我的列表和通知栏上能明显感知“正在逐步导入”，而不是长时间无结果。

## 非目标

1. 不新增独立的下载管理页或同步中心页。
2. 不实现 `OneDrive Graph delta` 增量同步。
3. 不引入代理域名、CDN 中转或服务端转码。
4. 不把缓存从原始文件改成 AAC 等转码缓存。
5. 不重做现有播放页 UI、歌单结构或媒体来源管理器整体布局。

## 当前问题

### 1. 首批歌曲出现时间仍然偏后

当前 `streamingSync` 已经支持分批提交，但前提仍是先完成 `enumerateSelection()` 并拿到完整 `Candidate[]`。这意味着：

1. 大目录遍历还没结束时，前面已经发现的歌曲无法先进入 hydration；
2. `OneDrive` 目录层级深、Graph 往返多时，这个等待尤其明显；
3. 用户体感仍像“先扫很久，再开始慢慢出来”。

### 2. 补全默认路径仍然过重

当前 provider 的 `hydrateCandidate()` 基本还是依赖下载临时文件读取 metadata。这样的问题是：

1. 网络高延迟来源上，单曲补全成本明显偏高；
2. 为了拿齐 `标题 + 歌手 + 时长`，很多歌曲被拖到重读取；
3. 本来应该先尽快可见的首批歌曲，被 metadata 读取成本拖慢。

### 3. 播放与同步的优先级虽然存在，但还不够显式

当前已经有“当前曲缓存 + 下一首预读”，但还存在两个可继续收敛的问题：

1. 同步正在跑时，预读不应该挤占首批歌曲进入列表的资源；
2. 当前播放曲目的缓存必须始终优先于同步重读取和下一首预读。

## 方案比较

### 方案 A：在现有二期框架上补流式快路径

做法：

1. 新增流式枚举接口，让 provider 可以边遍历边吐出 candidate；
2. `streamingSync` 优先消费流式产出，边产出边补全、边 ready 边提交；
3. 补全改成轻字段优先、重读取兜底；
4. UI 层只补状态反馈，不开新页面。

优点：

1. 直接命中“首批歌曲更快出现”的核心问题；
2. 保持 `OD / WebDAV / SMB` 共用一套框架；
3. 风险可控，属于二期框架的增量强化。

缺点：

1. 需要调整 provider 合约和 coordinator 控制流；
2. 测试要补足流式枚举与优先级行为。

### 方案 B：只调并发和 batch 阈值

做法：

1. 提高枚举和补全并发；
2. 缩小批量提交阈值；
3. 不改 provider 与 coordinator 接口。

优点：

1. 改动最小；
2. 容易快速落地。

缺点：

1. 仍然被“整轮枚举结束后才开始补全”的主瓶颈卡住；
2. 不能稳定保证首歌更快出现。

### 方案 C：只给 OneDrive 做特化优化

做法：

1. 单独为 `OneDrive` 写更激进的快路径；
2. 其他 provider 保持现状。

优点：

1. 对当前最慢来源见效快。

缺点：

1. 会把已有通用框架打裂；
2. 后续 `WebDAV / SMB` 还得重复补一遍。

### 结论

本次采用 **方案 A**。

原因：

1. 用户已经明确希望这套逻辑能通用，不接受只给 `OneDrive` 做一套特例；
2. 真实瓶颈不只是并发值，而是“枚举先整体完成”这个控制流；
3. 当前二期框架已经具备 batch commit、旧歌单保留和删除收尾机制，最合适的做法就是在此基础上补快路径，而不是另起炉灶。

## 核心设计

### 一、流式枚举接口

当前 provider 接口保留兼容：

1. `enumerateSelection(connection, selection) -> { complete, items }`

新增优先能力：

1. `streamEnumerateSelection(connection, selection, consumer)`

其中：

1. `consumer(batch)` 接收一批新发现的 `candidate`；
2. provider 可以按目录、按分页或按固定 batch 大小持续回调；
3. `streamingSync` 优先使用 `streamEnumerateSelection()`；
4. 若 provider 尚未实现，则回退到旧的 `enumerateSelection()` 一次性结果，保证兼容。

这意味着：

1. 前面的目录刚枚举出一批歌曲，就可以立即进入 hydration；
2. 不必等整棵目录树走完；
3. 老 provider 可以逐步迁移，不要求一次性全改。

### 二、单曲补全快路径

单曲补全在现有 `ready / degraded / committed` 状态机基础上增加更清晰的层级：

#### Level 0：轻字段直出

优先使用：

1. 远端枚举直接可得字段；
2. 文件名拆分规则；
3. 本地已有缓存记录或快照中可复用的轻字段。

只要这一级就拿齐 `标题 + 歌手 + 时长`，直接 `ready`。

#### Level 1：极轻量元数据

若 Level 0 不足，则使用 provider 暴露的最轻量 metadata 路径。要求：

1. 不整首下载；
2. 能用远端 API 直接字段就不读文件；
3. 能读最小必要片段就不下载完整文件。

#### Level 2：重读取兜底

只有前两级都不够时，才走现有“下载临时文件再读 metadata”的路径。

规则保持不变：

1. 满足 `标题 + 歌手 + 时长` 即进入 `ready`；
2. 同一轮累计 `3` 次后仍不满足，则进入 `degraded`；
3. `degraded` 仍允许按文件名歌名进入歌单；
4. 后续新的同步仍可把它补齐成正式 metadata。

### 三、流式提交

当前 `batchCommitter` 的双阈值规则保持不变：

1. 满 `10` 首 ready/degraded 即提交；
2. 或等待 `2` 秒超时提交，以先到者为准。

变化点在于：

1. 这些 ready/degraded 不再依赖“整轮枚举结束”；
2. 枚举一边进行，前面已经补好的歌曲就可以先进入提交器；
3. 因此用户会看到歌曲逐批进入“全部媒体”和目录歌单。

### 四、播放缓存与预读优先级

播放侧不改现有入口，只进一步明确任务优先级：

1. 当前实际播放曲目的缓存下载：最高；
2. 同步中的轻量枚举与轻量补全：中高；
3. 下一首预读：中；
4. 重读取补全：最低。

这样保证：

1. 当前播放始终不被后台任务拖慢；
2. 首批歌曲尽快进入列表不会被下一首预读反拖慢；
3. 下一首预读仍然保留，但属于可延后任务。

## UI 修改范围

本次只做必要可见层调整，不新增独立页面。

### 1. 媒体来源管理器

规则卡片和账号卡片补充更明确的同步状态：

1. `扫描中`
2. `排队中`
3. `已暂停，等待恢复`
4. `完成`
5. `失败`

保留现有手动更新入口，不增加新的复杂控制台视图。

### 2. 我的列表 / 生成歌单

用户可感知行为如下：

1. 更新开始时旧歌单继续显示，不清空；
2. 新补全成功的歌曲逐步追加到 `全部媒体` 和对应目录歌单；
3. 删除只在最后统一移除。

### 3. 通知栏

后台同步仍只通过通知栏反馈：

1. `扫描中`
2. `完成`
3. `失败`

通知内容以简洁进度为主，不新增主界面阻断弹窗。

### 4. 媒体来源页自动同步体验

自动同步仍只在：

1. 打开 App；
2. 进入媒体来源页

进行静默检查，并按 `1 天` 冷却规则排队，不额外增加干扰性弹窗。

## Provider 迁移策略

本次要求：

1. `onedrive` 优先实现 `streamEnumerateSelection()`；
2. `webdav` 与 `smb` 同步迁移到相同合约；
3. 若某 provider 暂未迁移，则仍可通过老 `enumerateSelection()` 工作，不阻塞整体发布。

这样可以保证：

1. 先把首歌更快出现的核心能力落下；
2. 同时不要求一次性推翻所有 provider 实现。

## 数据与状态规则

本次不新增新的正式索引表，继续复用：

1. `source_item`
2. `aggregate_song`
3. `generated playlists`
4. `media_cache`
5. `sync_run / sync_candidate / sync_snapshot`

状态规则不变：

1. `discovered / hydrating` 不能进入歌单；
2. `ready / degraded` 可以进入批量提交器；
3. `committed` 才是对用户正式可见的曲目；
4. 删除仍只在整轮 reconcile-delete 阶段统一处理；
5. 同步中途失败时，保留旧歌单和已提交的新歌，不执行删除阶段。

## 验证策略

实现完成后至少覆盖以下验证点：

1. `OneDrive` 大目录导入时，不需要等待整轮枚举结束，第一批歌曲即可进入歌单。
2. `WebDAV / SMB` 走同一套流式快路径或兼容回退路径，不破坏现有行为。
3. 更新开始时旧歌单不清空。
4. 更新过程中歌曲逐步追加。
5. 删除只在最后统一处理。
6. 当前播放曲目缓存优先，不被预读或重读取补全拖慢。
7. 下一首预读仍然有效，但在同步繁忙时属于次优先级任务。
8. 媒体来源管理器、我的列表和通知栏上的状态反馈与实际同步状态一致。

## 范围结论

这次实现的本质不是“再做一版新的远端同步”，而是给现有二期框架补上真正的快路径：

1. 让歌曲更早出现；
2. 让轻量补全优先；
3. 让播放缓存与预读和同步优先级更一致；
4. 让用户在现有 UI 中就能明显感知“正在逐步导入”。

它保持当前统一架构，也为后续再做 `OneDrive delta`、代理优化或更细粒度缓存策略预留空间。
