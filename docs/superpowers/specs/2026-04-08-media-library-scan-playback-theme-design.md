# 媒体库扫描调度、播放页主题与远端缓存跳播设计

日期：2026-04-08  
状态：已确认，待用户复核后进入实现规划

## 概述

本设计同时处理一组已经被用户明确确认、且彼此有关联的问题：

1. 远端媒体来源扫描在多入口、多 runtime 场景下会互相踩状态，导致生成歌单被错误刷空。
2. 播放页在动态背景下的前景可读性不稳定，顶部系统状态栏在播放页也需要强制白色。
3. 深色主题下歌词高亮不够明确，需要把当前行默认提高到白色层级。
4. 现有主题体系需要开放更细粒度的颜色自定义，并区分浅色 / 深色两套配置。
5. OneDrive 的用户可见简称改为 `OD`。
6. 远端歌曲首次播放、尚未完整缓存时，拖动进度条后 UI 虽响应，但实际播放不会跳到目标时间点。

本次不做新的媒体来源类型，不重做整个主题系统，也不把远端歌曲缓存改成转码缓存。目标是先把扫描稳定性、播放可读性和远端首次播放交互收敛到一套能持续扩展的行为模型。

## 目标

1. 让远端扫描在存在并发触发时，不再把生成歌单错误刷空。
2. 让用户在“已有扫描正在运行”时，明确选择“继续之前的扫描”或“当前扫描优先”。
3. 让播放页在各种封面亮度下都保持稳定可读，并让播放页顶部系统状态栏固定为白色。
4. 让浅色 / 深色主题分别拥有独立的主题色与歌词颜色配置。
5. 让远端歌曲首次播放、未完整缓存时，用户拖动进度条后能进入“等待该时间点加载并从该点播放”的状态。

## 非目标

1. 不重做全局主题 token 架构。
2. 不把远端歌曲缓存改成转码缓存，也不引入 AAC 转码链路。
3. 不做 OneDrive / WebDAV / SMB 的新 provider 接入。
4. 不把扫描冲突处理做成全局固定偏好；本次采用按次询问。
5. 不实现“边拖边精准 seek 到任意尚未拉取字节范围”的完整流式播放器重构。

## 方案选择

针对扫描冲突，有三类方案：

### 方案 A：只修当前 bug

做法：

1. 仅修复当前 `running -> queued` 误恢复问题。
2. 不引入暂停态，只做最小补丁。

优点：

1. 改动小。
2. 可以快速止血。

缺点：

1. 只能修当前已知问题。
2. 无法满足“当前扫描优先，旧任务回队列”的明确交互需求。

### 方案 B：增加显式扫描协调器与暂停态

做法：

1. 在现有导入队列之上增加显式扫描协调规则。
2. 新增 `paused` / `resumable` 语义。
3. 冲突时弹出选择，由用户决定继续旧任务还是当前任务优先。
4. 生成歌单默认显示上次成功快照，直到本次完整扫描成功后再整体切换。

优点：

1. 能同时解决状态互踩、歌单刷空和交互决策问题。
2. 后续对 OD / WebDAV / SMB 都能复用。

缺点：

1. 改动面大于单点补丁。

### 方案 C：强制所有远端扫描串行

做法：

1. 所有远端扫描只允许排队，不允许用户选择。

优点：

1. 状态机最简单。

缺点：

1. 不符合已确认的产品要求。

### 结论

采用 **方案 B**。

原因：

1. 用户已经明确要求在冲突时自己选择“继续之前”还是“当前优先”。
2. 当前问题不是单个 bug，而是扫描任务所有权和恢复机制设计不完整。
3. 只有显式状态机和快照保护，才能避免生成歌单在中途被错误重写成空列表。

## 一、扫描调度与空列表保护

### 1. 当前问题

当前远端扫描存在两个根问题：

1. 运行中的任务在新的 runtime 初始化队列时，可能被误判为“需要恢复的僵尸任务”，从 `running` 直接改回 `queued`。
2. 生成歌单在扫描进行中就会基于当前快照被重算并写回，一旦快照尚未稳定，就可能把已有内容覆盖成空列表或半成品。

用户的实际复现已经能说明这个问题：

1. OD 正在扫描。
2. 用户新建 WebDAV 并开始扫描。
3. 再回到“我的列表”时，OD 生成歌单被刷空。

这说明“新增另一个来源”并不是根因，根因是扫描状态与歌单可见状态之间没有被隔离。

### 2. 任务状态模型

远端扫描任务统一使用以下显式状态：

1. `queued`
2. `running`
3. `paused`
4. `success`
5. `failed`
6. `canceled`

其中：

1. `paused` 表示任务被用户让位，不是失败，也不是丢失。
2. `running` 只能属于一个明确的 runtime owner。
3. `queued` 只表示等待执行，不能再兼任“原来正在运行，但现在我懒得区分”的恢复态。

### 3. 任务所有权

每个进入 `running` 的远端扫描任务都需要带一个运行时所有权标识，例如：

1. `runtimeOwnerId`
2. `startedAt`
3. `heartbeatAt`

设计要求：

1. 新 runtime 初始化时，不能无脑把所有 `running` 改回 `queued`。
2. 只有在明确判断为“前一个 owner 已经失联”的情况下，才允许恢复任务。
3. 若任务仍被活跃 owner 占用，则当前 runtime 只读取状态，不接管执行权。

这样可以解决“后台 runtime 还在扫，前台 runtime 又把它当死任务重排”的问题。

### 4. 扫描冲突交互

当已有远端扫描正在执行，用户又发起新的远端扫描时：

弹出冲突对话框，提供两个明确选项：

1. `继续之前的扫描`
2. `当前扫描优先`

行为如下：

#### 继续之前的扫描

1. 当前运行中的任务保持 `running`。
2. 新任务进入 `queued`。
3. 当前任务完成后，新任务再执行。

#### 当前扫描优先

1. 当前运行中的任务改为 `paused`。
2. 新任务立即变为 `running`。
3. 新任务结束后，被暂停的旧任务自动回到队列并恢复执行。

本次不提供“直接丢弃之前扫描”的选项，因为用户明确要求旧任务要自动回队列继续。

### 5. 生成歌单可见状态保护

远端规则生成歌单默认遵循 **上次成功快照保护**：

1. 若某条规则已有成功快照：
   1. 新一轮扫描开始后，歌单继续显示上次成功结果；
   2. 本轮扫描中产生的中间数据只能作为内部工作区数据，不直接覆盖用户可见歌单；
   3. 只有当本轮扫描完整成功后，才整体切换到新结果。
2. 若某条规则从未成功过：
   1. 可以显示空歌单占位；
   2. 同时显示“正在导入 / 正在扫描”的状态和进度摘要。

这条规则是本次避免“歌单中途刷空”的核心。

### 6. 失败与暂停时的显示

#### 扫描失败

1. 保留上次成功歌单内容。
2. 更新状态摘要、失败原因、上次成功时间。
3. 不允许把失败结果写成空歌单覆盖旧内容。

#### 扫描暂停

1. 保留当前用户可见歌单内容。
2. 在规则卡片和账号卡片上显示“已暂停，等待恢复”。
3. 不做列表清空，也不把暂停等价成失败。

### 7. 应用重启恢复

应用重启后，远端扫描恢复规则如下：

1. 若发现 `running` 任务仍归属于活跃 owner：
   1. 不接管；
   2. 只展示该任务为运行中。
2. 若发现 `running` 任务 owner 已失联：
   1. 将其改为可恢复状态；
   2. 由新的协调器重新放回队列继续执行。
3. 已处于 `paused` 的任务不直接执行；
   1. 先恢复当前优先级更高的活动任务；
   2. 再按顺序恢复被暂停任务。

## 二、远端首次播放未缓存完成时的跳播

### 1. 当前问题

当前远端歌曲首次播放时，如果本地缓存只完成了前 20 秒左右：

1. 用户把进度拖到 2 分钟；
2. UI 进度会更新；
3. 但实际音频仍从 20 秒继续播放。

这会造成“界面时间”和“真实音频时间”不一致。

### 2. 目标行为

当远端歌曲尚未完整缓存，用户拖动到任意目标时间点时：

1. 界面先响应到目标时间点；
2. 播放状态进入“等待该时间点加载”；
3. 当前音频输出不再按旧位置继续推进；
4. 一旦数据可用，从目标时间点开始播放。

也就是说：

1. 不禁止用户拖动。
2. 不要求在未就绪时立即播放成功。
3. 但不能继续假装还在旧时间点正常播放。

### 3. 行为模型

新增远端播放的“待跳转目标”概念，例如：

1. `pendingSeekTime`
2. `seekAwaitingData`

当用户在远端首次播放阶段拖动进度时：

1. 记录新的 `pendingSeekTime`。
2. UI 进度与当前时间文案立即跳到目标值。
3. 播放器进入“等待该时间点数据”的加载状态。
4. 当缓存覆盖到可播放范围后，执行真正 seek，并从目标时间继续播放。

如果用户在等待过程中再次拖动：

1. 新目标覆盖旧目标；
2. 只保留最后一次 seek 意图。

### 4. 首次播放与已缓存播放的区别

#### 首次远端播放

1. 允许用户拖动到任意目标时间。
2. 若目标时间点尚未可播，则进入等待加载状态。

#### 已完整缓存或目标范围已可播

1. 正常立即 seek。
2. 不再走等待加载流程。

### 5. 失败回退

若目标时间点在合理等待后仍不可播放：

1. 保持加载提示；
2. 提供失败回退到最近可播放点或重新请求资源；
3. 不能默默继续从旧时间点播放并让 UI 停留在新时间点。

## 三、播放页视觉与状态栏

### 1. 背景基线

播放页继续使用当前歌曲封面做模糊背景，但统一新增一层稳定的深灰遮罩：

1. 目的是把过亮、过杂的封面收敛到可读范围。
2. 不要求压成纯黑。
3. 重点是让前景文字和图标对比度稳定。

### 2. 状态栏

播放页顶部系统状态栏采用固定白色策略：

1. 时间、电量、信号等系统字体 / 图标统一为白色。
2. 该规则只在播放详情页生效。
3. 离开播放详情页后，恢复正常页面跟随主题的状态栏逻辑。

### 3. 前景层

播放页正文与主要图标采用高对比前景规则：

1. 主信息使用高对比白。
2. 次级信息使用弱一档的近白。
3. 不再用灰色辅助 token 承担播放页关键正文。

## 四、歌词颜色与主题自定义

### 1. 浅色 / 深色两套配置

颜色配置按主题模式拆成两套：

1. 浅色主题配置
2. 深色主题配置

两套配置相互独立保存、独立读取。

### 2. 自定义项

每套至少开放以下颜色项：

1. 主题主色
2. 歌词当前行
3. 歌词未播放行
4. 歌词翻译
5. 歌词罗马音

本次仍然挂在现有主题体系中，不新增播放页专属主题系统。

### 3. 默认规则

深色主题下：

1. 歌词当前行默认白色。
2. 未播放行、翻译歌词、罗马音使用分层浅灰。

浅色主题下：

1. 当前行默认使用主题强调色。
2. 其他歌词层级依次减弱。

### 4. 设置入口

在现有主题设置内增加“颜色自定义”入口：

1. 浅色主题一组
2. 深色主题一组

不新增新的独立导航页。

## 五、来源简称统一

内部 providerType 仍保持 `onedrive`，但所有用户可见简称改为：

1. `OD`

适用范围包括：

1. 歌单来源标签
2. 歌曲列表来源徽标
3. 与 OneDrive 简称相关的轻量展示位

## 六、实现边界

本次建议拆成两个实现工作流：

### A. 稳定性修复

包含：

1. 扫描任务所有权
2. `paused` 状态
3. 冲突弹框
4. 上次成功快照保护
5. 远端首次播放未缓存完成时的跳播等待
6. `OD` 标签替换

### B. 视觉与主题

包含：

1. 播放页深灰遮罩
2. 播放页顶部状态栏白字
3. 深色主题当前歌词白色
4. 浅色 / 深色两套颜色配置
5. 歌词颜色自定义入口与读取逻辑

拆开的原因是：

1. 稳定性问题需要先止血。
2. 视觉改动不应与扫描状态机耦合。

## 七、验证策略

### 1. 调度与快照保护

至少验证：

1. 有远端任务运行时，再发起新任务会出现冲突选择。
2. 选择“继续之前的扫描”时，新任务入队，旧任务继续。
3. 选择“当前扫描优先”时，旧任务进入 `paused`，新任务先跑，旧任务随后恢复。
4. 扫描中、失败、暂停时，已有生成歌单不会被刷空。

### 2. 远端首次播放跳播

至少验证：

1. 远端首次播放、未完整缓存时，拖动到新时间点后 UI 会进入等待加载状态。
2. 等缓存覆盖到目标时间点后，会从目标时间开始播放。
3. 不再出现 UI 已跳到 2 分钟、音频却从 20 秒继续播放的情况。

### 3. 视觉回归

至少验证：

1. 播放页顶部系统状态栏为白色。
2. 播放页深灰遮罩存在且保证白字可读。
3. 深色主题下歌词当前行默认白色。
4. `OD` 简称在预期位置生效。
5. 浅色 / 深色两套主题色与歌词色配置能分别生效。

## 自检

### 占位检查

1. 无 `TODO`、`TBD`、`稍后再说` 之类占位语。
2. 冲突选择、快照保护、首次跳播的行为都已明确。

### 一致性检查

1. 扫描冲突策略与“用户自己选择先执行谁”一致。
2. 快照保护与“不要把歌单中途刷空”一致。
3. 跳播等待与“点了就跟着走，等加载到再从那里播”一致。

### 范围检查

1. 本次没有扩展到转码缓存或新 provider。
2. 本次保持在扫描稳定性、播放页视觉和远端首次播放交互范围内。
