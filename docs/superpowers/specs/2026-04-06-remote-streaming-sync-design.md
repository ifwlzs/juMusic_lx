# 二期远端流式同步与播放优化设计

日期：2026-04-06  
状态：已确认，可进入实现规划

## 概述

一期已经完成 `本地目录`、`WebDAV`、`SMB` 的统一媒体曲库接入，并把 `OneDrive` 作为后续扩展能力接入到同一套媒体来源框架中。当前远端来源的核心问题已经从“能不能接入”转成“接入后好不好用”。

当前远端来源，尤其是 `OneDrive`，在体验上仍有两个明显短板：

1. 一次更新往往要等整批扫描和整批元数据读取结束后，歌曲才会统一出现在歌单里；
2. 扫描、播放缓存、后台更新、预读之间还没有形成一套统一的优先级和任务模型。

二期不再把重点放在“再新增一个来源类型”，而是补一层通用的远端流式同步框架，让 `OneDrive`、`WebDAV`、`SMB` 都能用同一套“先快扫、再补全、分批入库、最后判删”的模型工作。这样既能解决当前 `OneDrive` 的等待问题，也能把现有 NAS 来源一起拉到同一条更稳的路径上。

## 目标

1. 为 `OneDrive`、`WebDAV`、`SMB` 建立统一的远端流式同步框架。
2. 把当前“整批扫完再统一入库”改成“快速枚举 + 分级补全 + 分批提交 + 收尾判删”。
3. 让歌曲在满足 `标题 + 歌手 + 时长` 后即可进入歌单，而不是等整批扫描结束。
4. 更新期间保留旧歌单，新补全成功的歌曲逐步追加，最后再统一判断源端删除。
5. 对同一轮同步内累计完成 `3` 轮补全仍失败的歌曲，允许降级成“文件名歌名”进入歌单，避免长时间卡住。
6. 在打开 App 或进入媒体来源页时支持自动同步检查，并通过 `1 天` 冷却时间控制频率。
7. 后台任务只通过通知栏显示 `扫描中 / 完成 / 失败`，不打断主界面操作。
8. 播放远端歌曲时继续沿用业务层缓存，并补上“缓存当前曲目 + 预读下一首”的统一策略。
9. 为后续 `Emby`、`Navidrome` 或 `OneDrive delta` 优化预留通用同步接口。

## 非目标

1. 不在二期实现独立的“离线下载管理”入口或下载中心页面。
2. 不在二期实现 `OneDrive Graph delta` 增量同步。
3. 不在二期实现多 `OneDrive` 企业账号并发使用。
4. 不在二期接入 `Emby` 或 `Navidrome` provider 本体。
5. 不在二期实现用户资料本地与远端同步。
6. 不在二期实现歌词时间偏移校准 UI、偏移持久化入口或其他歌词精调功能。

## 当前问题归纳

### 现有同步模型的问题

1. 远端来源的更新仍偏向“整批扫描完成后再统一落库”。
2. 单曲是否可见、是否可播，过度依赖整轮扫描是否结束，而不是依赖该单曲本身是否已经补全。
3. 元数据读取成本高时，会把用户的可见结果整体向后推迟。
4. 更新过程中如果过早处理删除，歌单会出现“新歌还没补完，旧歌先不见了”的抖动体验。

### 现有后台任务的问题

1. 扫描、补元数据、播放缓存和预读没有统一任务优先级。
2. 自动同步触发策略还不完整，无法在“尽量自动”和“不要过度打扰”之间取得平衡。
3. 通知栏展示还没有直接围绕媒体来源后台更新建模。

## 方案比较

### 方案 A：通用流式补全入库框架

1. 先快速枚举远端候选文件。
2. 再逐首做分级补全。
3. 某首歌一旦满足入库标准，就进入歌单并立刻可播。
4. 最后统一处理删除。

优点：

1. 最贴合“先可播、再补全”的目标。
2. 可以同时覆盖 `OneDrive`、`WebDAV`、`SMB`。
3. 自动同步、通知栏进度和预读策略都有统一落点。

缺点：

1. 需要把当前远端扫描提交模型改成增量式。
2. 仓储、歌单重建和收尾快照逻辑会明显复杂一些。

### 方案 B：保留整批提交模型，只优化元数据读取速度

1. 不改导入事务模型。
2. 只优化读取方式和并发配置。

优点：

1. 改动更小。
2. 风险更低。

缺点：

1. 无法解决“整批扫完才能看见歌”的根问题。
2. 只能改善等待时间，不能改变等待模型。

### 方案 C：先补自动同步和预读，不改扫描模型

1. 优先做后台自动更新。
2. 优先做当前缓存和下一首预读。
3. 扫描模型维持现状。

优点：

1. 实现最快。
2. 容易独立交付。

缺点：

1. 不能解决当前最痛的同步等待问题。
2. 只能治标，不能治本。

### 结论

二期采用 **方案 A**。

原因很直接：本轮目标不只是“让远端再快一点”，而是“让远端歌曲在扫描过程中逐步可见、逐步可播”。只优化元数据读取速度无法满足这个要求，必须把同步模型本身改成流式。

## 二期已确认的产品规则

1. `OneDrive`、`WebDAV`、`SMB` 三个远端来源统一迁到同一套流式同步框架。
2. 歌曲默认只有在满足 `标题 + 歌手 + 时长` 后才允许进入歌单。
3. 如果一首歌在同一轮同步内累计完成 `3` 轮补全仍失败，则允许降级成“文件名歌名”进入歌单。
4. 更新开始时保留旧歌单，不允许一开始就清空。
5. 更新过程中，新补全成功的歌曲逐步追加到“全部媒体”和目录歌单。
6. 删除只在整次任务收尾阶段统一判断，不提前处理。
7. 自动同步触发点为：打开 App、进入媒体来源页。
8. 自动同步冷却时间为 `1 天`。
9. 后台更新只通过通知栏展示，不额外打断主界面。
10. 远端播放时缓存当前曲目，并预读下一首。
11. 下一首预读在任何网络环境都启用。
12. 批量提交采用双阈值策略：`满 10 首 ready 即提交，或等待 2 秒超时即提交，以先到者为准`。
13. 同一首歌在同一轮同步内最多允许 `3` 轮补全，仍不满足入库标准则降级入库。

## 总体架构

二期不新增第二套曲库域，而是在现有媒体曲库域上补一层通用的 `Remote Streaming Sync` 框架。正式曲库索引仍然是唯一事实源，远端流式同步只是在正式写入前增加一层同步工作区和任务编排。

### 1. `Remote Provider`

远端 provider 继续只负责来源差异，不承担业务层歌单逻辑。二期下的 provider 统一需要提供：

1. 轻量枚举目录和候选音频文件；
2. 返回稳定的文件身份和版本标识；
3. 支持分级元数据补全所需的最便宜读取方式；
4. 支持播放时缓存下载。

Provider 不负责：

1. 何时把歌曲放进歌单；
2. 删除何时生效；
3. 批量提交节流；
4. 自动同步调度；
5. 播放预读优先级。

### 2. `Streaming Sync Coordinator`

新增一个通用的同步协调层，负责把远端 provider 输出组织成统一流程。它负责：

1. 创建和维护同步任务；
2. 调度枚举阶段、补全阶段、批量提交阶段和收尾判删阶段；
3. 控制不同阶段的并发和优先级；
4. 维护通知栏可见状态；
5. 把自动同步和手动同步统一到同一条任务链路上。

### 3. `Hydration Queue`

每首候选歌曲进入独立的补全状态机。该队列负责：

1. 逐首补全歌曲元数据；
2. 跟踪尝试次数、最后错误和当前状态；
3. 在满足入库条件后把歌曲送入 ready 队列；
4. 在同一轮同步内累计完成 `3` 轮补全仍失败后，把歌曲降级成文件名入库候选。

### 4. `Ready Batch Committer`

为了避免“每补完一首就重建一次歌单”，ready 项目不会逐条直接落正式索引，而是进入批量提交器。它负责：

1. 收集已经 `ready` 或 `degraded` 的歌曲；
2. 满 `10` 首或等待 `2` 秒时触发一次批量写入；
3. 批量更新 `source_item`、`aggregate_song` 和生成歌单；
4. 批量刷新通知栏进度，而不是每首都刷新一次。

### 5. `Delete Reconciler`

删除永远放在最后。它负责：

1. 对比本次任务确认过的来源快照和历史快照；
2. 只删除本轮已确认不再存在的歌曲；
3. 避免扫描进行到一半时提前移除旧歌。

### 6. `Playback Cache Scheduler`

播放侧继续复用现有缓存管理器，但增加统一的预读调度层。它负责：

1. 当前曲目缓存优先级最高；
2. 当前曲目稳定播放后，后台预读下一首；
3. 如果下一首已有有效缓存，则直接命中；
4. 当前曲目切换时，取消或降级旧预读任务。

## 同步主流程

一次远端同步任务统一拆为 4 个阶段。

### 阶段 A：`enumerate`

目标是尽快拿到规则范围内的候选文件清单，而不是立刻把所有元数据读满。

这一阶段只做：

1. 枚举目录树；
2. 过滤音频文件；
3. 产出轻量候选项；
4. 记录稳定文件身份、路径、大小、修改时间、版本标识等轻字段。

这一阶段不做：

1. 整首下载读元数据；
2. 正式写入歌单；
3. 删除旧数据。

### 阶段 B：`hydrate`

候选文件进入逐首补全队列。每首歌独立推进，不允许一首卡住整次任务。

补全过程遵循分级策略：

1. 先用远端已有字段和文件名规则做轻量补全；
2. 不足时再做小片段读取；
3. 还不足时才走更重的完整读取；
4. 达到 `标题 + 歌手 + 时长` 即进入 `ready`；
5. 同一轮同步累计完成 `3` 轮补全仍不满足条件，则进入 `degraded`。

### 阶段 C：`commit`

`ready` 或 `degraded` 的歌曲进入 ready 批量提交器。提交器以批次方式更新正式索引：

1. 每凑够 `10` 首立即提交；
2. 如果一直凑不满，则等待 `2` 秒后强制提交；
3. 提交完成后，这些歌曲立刻出现在“全部媒体”和对应目录歌单里；
4. 对用户表现为“歌曲持续出现”，而不是“一次性全部出现”。

### 阶段 D：`reconcile-delete`

只有当前 3 个阶段完成后，才开始处理删除：

1. 对比本次确认的来源快照与历史快照；
2. 找出源端已删除的歌曲；
3. 从正式索引和生成歌单中统一移除；
4. 更新新的历史快照。

如果本次同步中途失败，删除阶段不执行。

## 单曲状态机

每首候选歌曲都遵循同一套状态机：

1. `discovered`：已枚举到，但还没开始补全；
2. `hydrating`：正在补全元数据；
3. `ready`：满足 `标题 + 歌手 + 时长`；
4. `degraded`：同一轮同步内累计完成 `3` 轮补全仍失败，降级成文件名歌名；
5. `committed`：已写入正式索引，已出现在歌单中；
6. `dropped`：确认不是音频、无效、重复或在收尾阶段被判定删除。

状态规则：

1. `discovered` 和 `hydrating` 不能进入歌单；
2. `ready` 和 `degraded` 进入批量提交队列；
3. `committed` 才是对用户可见的正式状态；
4. `dropped` 不参与后续展示和播放。

## 分级补全策略

三类远端来源统一使用同一套补全优先级，不再由某个 provider 独占特例逻辑。

### Level 1：轻量补全

优先使用：

1. 远端目录枚举本身可提供的字段；
2. 文件名规则；
3. 已有轻量元数据缓存。

如果在这一级已经得到 `标题 + 歌手 + 时长`，则直接进入 `ready`。

### Level 2：小片段补全

若 Level 1 不足，则进一步读取最小必要文件片段。目标是：

1. 尽量不整首下载；
2. 只读取对标签提取真正必要的部分；
3. 由 provider 暴露“最便宜的元数据读取方式”。

### Level 3：重读取兜底

若前两级仍不足，则走更重的完整读取或临时下载。

这一层的并发必须最低，只作为兜底，不作为主路径。

### 补全失败降级

如果同一首歌在当前同步中累计完成 `3` 轮补全仍无法满足 `标题 + 歌手 + 时长`：

1. 先记录错误和尝试次数；
2. 达到 `3` 轮补全阈值后进入 `degraded`；
3. 允许用文件名歌名入库；
4. 后续新的手动或自动同步仍可再次尝试把它补成正式元数据。

## 并发与优先级控制

二期需要明确分离不同任务类型，而不是让所有网络与 I/O 任务平铺竞争。

### 同步并发分层

1. 枚举阶段并发最高；
2. 轻量补全并发中等；
3. 重读取补全并发最低。

### 播放与同步的任务优先级

1. 当前正在播放曲目的缓存下载：最高；
2. 同步中的轻量枚举和轻量补全：中高；
3. 下一首预读：中；
4. 重读取补全：低。

这样可以保证：

1. 正在听的歌永远优先；
2. 同步不会因为预读被完全拖慢；
3. 预读也不会把同步任务全部饿死。

## 自动同步设计

### 触发点

自动同步只在两个入口触发检查：

1. 打开 App；
2. 进入媒体来源页。

### 冷却策略

1. 距离上次成功同步不足 `1 天` 时，不起同步任务；
2. 超过 `1 天` 时，自动创建后台同步任务；
3. 手动更新永远不受冷却时间限制。

### 作用范围

自动同步只作用于：

1. 已启用的远端连接；
2. 至少存在一条导入规则的连接；
3. 当前没有同连接重复任务在跑的连接。

## 通知栏展示设计

后台同步只通过通知栏反馈，不额外打断主界面。

### `扫描中`

建议展示：

1. 来源名或规则名；
2. 已枚举数量；
3. 已补全数量；
4. 失败数量；
5. 当前阶段。

### `完成`

建议展示：

1. 本次新增数量；
2. 删除数量；
3. 降级数量；
4. 失败数量。

### `失败`

建议展示：

1. 来源名；
2. 简短错误原因；
3. 是否仍保留旧歌单；
4. 可继续手动重试的状态。

## 播放缓存与下一首预读

二期不做独立离线入口，但把“播放缓存”做成更明确的行为。

### 当前曲目

1. 用户实际播放远端歌曲时，优先保证当前曲目缓存可用；
2. 如果已有有效缓存，则直接命中；
3. 如果缓存失效，则立即删除并重新下载。

### 下一首预读

1. 当前曲目进入稳定播放后，后台预读下一首；
2. 下一首预读在任何网络环境都启用；
3. 如果下一首已有有效缓存，不重复下载；
4. 如果切歌，则旧预读任务取消或降级。

### 失败处理

1. 当前曲目缓存失败时，立即重试一次；
2. 仍失败则走现有远端兜底播放路径；
3. 下一首预读失败只记录状态，不打断当前播放；
4. 真正播到下一首时，再以“当前曲目”优先级重新拉取。

## 异常处理

### 同步中途失败

1. 保留旧歌单；
2. 已经成功提交的新歌保留；
3. 删除阶段不执行；
4. 通知栏标记失败并附简短原因。

### 单曲补全失败

1. 记录尝试次数和最后错误；
2. 当前轮累计完成 `3` 轮补全后仍失败则降级入库；
3. 后续新的同步仍可再次尝试补齐。

### 预读失败

1. 不影响当前播放；
2. 下次真正播放时按正常缓存流程重试；
3. 不把预读失败误判成歌曲不可播放。

## 二期后的预留项

以下能力明确不进入当前二期，但设计时要预留接口：

1. `OneDrive Graph delta` 增量同步；
2. 独立离线下载和下载管理页；
3. `Emby`、`Navidrome` provider；
4. 多企业账号并发；
5. 用户资料远端同步；
6. 歌词偏移校准和偏移持久化入口。

## 验证策略

二期实现完成后，至少要覆盖以下验证点：

1. `OneDrive`、`WebDAV`、`SMB` 都能走同一套流式同步主流程。
2. 某首歌补全成功后，无需等待整轮扫描结束，就能出现在歌单里。
3. 未达到 `标题 + 歌手 + 时长` 标准的歌曲不会提前进入正式歌单。
4. 达到 `3` 轮补全阈值后，歌曲会以“文件名歌名”降级进入歌单，而不是一直不可见。
5. 更新开始时旧歌单保持可见。
6. 更新过程中新增歌曲逐步追加。
7. 删除只在收尾阶段统一处理。
8. 自动同步在打开 App 和进入媒体来源页时按 `1 天` 冷却规则触发。
9. 后台同步的状态只出现在通知栏。
10. 当前曲目缓存与下一首预读不会打断正常播放。

## 实现建议

为了降低风险，建议按以下顺序推进：

1. 先补同步工作区、状态机和 ready 批量提交器；
2. 再把 `OneDrive` 接到通用流式同步链路；
3. 然后把 `WebDAV` 和 `SMB` 迁到同一模型；
4. 再补收尾判删；
5. 再补自动同步和通知栏任务；
6. 最后补“当前曲目缓存 + 下一首预读”以及相应调优。

## 结论

二期的核心不是“继续把更多来源接进来”，而是把远端来源从“整批同步思维”升级到“流式同步思维”。只要这层模型建立起来，`OneDrive` 当前的等待问题、`WebDAV/SMB` 的一致性问题，以及后续新 provider 的接入成本，都会一起下降。

本设计因此选择优先交付一套通用的远端流式同步框架，并以 `OneDrive` 为第一验证对象，同时让 `WebDAV`、`SMB` 在同一二期内完成迁移。
