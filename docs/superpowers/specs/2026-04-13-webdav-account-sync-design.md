# WebDAV 手动上传账号同步设计

日期：2026-04-13  
状态：已确认，待用户审阅

## 概述

当前应用已经具备：

1. “设置 -> 备份与恢复”中的本地导入 / 导出能力；
2. 媒体来源 `WebDAV / SMB / OneDrive / 本地目录` 的连接与规则管理；
3. 播放历史与年终统计相关的导出与缓存能力。

但如果用户想把“当前设备上的账号配置”同步到自己的云端目录，当前仍有明显缺口：

1. 没有独立于媒体来源的“同步账号”配置；
2. 没有把“设置 + 媒体来源连接 / 凭据 / 规则”打包上传到远端的能力；
3. 没有在包含敏感信息时对同步文件进行密码加密。

本设计新增一个首版“账号同步”能力，采用：

1. **手动上传**；
2. **WebDAV 独立同步账号**；
3. **同步范围只包含设置与媒体来源配置**；
4. **同步包包含敏感信息，但使用用户输入的同步密码加密后再上传**。

目标是先把“可上传、可追溯、可为后续恢复铺路”的同步基础打稳，而不是在首版就把自动同步、冲突合并、多版本历史和下载恢复全部一起做完。

## 目标

1. 在“设置 -> 备份与恢复”中新增独立的“账号同步”区块。
2. 提供一组**独立于媒体来源**的 WebDAV 同步账号配置。
3. 支持用户手动验证同步账号连接。
4. 支持用户手动触发一次上传同步。
5. 上传内容包含：
   1. 当前应用设置；
   2. 媒体来源连接；
   3. 媒体来源凭据；
   4. 导入规则。
6. 上传前要求用户输入“同步密码”，并用该密码加密同步包后再上传到 WebDAV。
7. 同步密码**不持久化保存**在本地。
8. 上传固定覆盖远端“最新同步文件”，为后续下载恢复保留格式兼容空间。

## 非目标

1. 本次不做实时同步、后台定时同步或应用启动自动同步。
2. 本次不做下载恢复。
3. 本次不做多设备冲突检测。
4. 本次不做远端历史版本保留或回滚。
5. 本次不同步播放历史、年终统计缓存、歌单、下载缓存或播放器运行态。
6. 本次不复用媒体来源里的 WebDAV 连接作为同步目标。
7. 本次不把同步密码保存在设置或媒体来源仓库中。

## 产品规则

1. “账号同步”属于低频管理能力，入口放在“设置 -> 备份与恢复”，不放在媒体来源主流程中。
2. 同步账号与媒体来源账号是两套独立模型：
   1. 媒体来源账号负责听歌、扫描与播放；
   2. 同步账号负责把配置文件上传到用户指定的 WebDAV 目录。
3. 用户修改同步账号任一字段后，之前的“已验证”状态立即失效，必须重新验证后才能上传。
4. 上传同步时必须再次输入同步密码与确认密码；同步密码只在本次上传过程中驻留内存。
5. 同步包中允许包含媒体来源凭据及其他持久化敏感设置，但远端只能保存**加密后的同步文件**，不能上传明文配置。
6. 首版远端只保留一份“最新同步文件”，后一次上传会覆盖前一次结果。
7. 远端目录若不存在，应用会在上传时尝试自动创建；已存在则直接复用。

## 入口与页面结构

### 1. 备份与恢复页新增区块

在现有“列表数据”“所有数据”“播放历史”之后新增一个独立分组：

1. 标题：`账号同步`
2. 按钮：`配置 WebDAV`
3. 按钮：`上传同步`

该区块同时展示轻量状态摘要：

1. 当前同步账号名称；
2. 远端目录；
3. 最近验证时间；
4. 最近上传时间；
5. 最近上传结果摘要。

### 2. 配置 WebDAV 弹窗

点击“配置 WebDAV”后，弹出独立配置弹窗，字段包括：

1. `displayName`：本地显示名称；
2. `serverUrl`：WebDAV 服务地址；
3. `username`；
4. `password`；
5. `remoteDir`：远端同步目录。

操作按钮：

1. `测试连接`
2. `保存`
3. `取消`

规则：

1. 只有“测试连接”成功后才允许保存；
2. 保存后回到备份页；
3. 编辑已有账号时复用同一弹窗；
4. 账号字段发生变化后，已验证状态失效。

### 3. 上传同步弹窗

点击“上传同步”后，不直接上传，而是先弹出“同步密码”输入框，包含：

1. `同步密码`
2. `确认同步密码`

同时展示说明文案：

1. 同步密码仅用于本次加密上传；
2. 应用不会在本地保存该密码。

输入合法后才允许开始上传。

## 数据范围与同步内容

### 1. 同步范围

首版只同步两大类内容：

1. `settings`
2. `mediaSource`

其中：

### `settings`

表示当前应用的持久化设置快照。

导出来源应与当前设置初始化 / 保存链路保持一致，确保未来下载恢复时可以直接走同一份设置结构。

### `mediaSource`

只保留能重建媒体来源账号与导入规则的最小集合：

1. `connections`
2. `credentials`
3. `importRules`

### 2. 明确排除内容

以下内容首版不得进入账号同步包：

1. `sourceItems`
2. `aggregateSongs`
3. `playStats`
4. `playHistory`
5. 年终统计缓存（`yearSummary / yearTimeStats / yearEntityStats / lifetimeEntityIndex`）
6. `importSnapshots`
7. `syncSnapshots`
8. 同步运行态 / job 队列
9. 歌单与列表数据
10. 下载缓存、本地临时文件、播放器当前运行态
11. 同步账号自身的 `profile / lastValidatedAt / lastUploadAt / lastUploadStatus / lastUploadMessage`

原因：

1. 这些数据体积更大且容易过时；
2. 它们不是“账号恢复”首版必须数据；
3. 部分数据可以通过后续重新扫描或重新生成恢复；
4. 把它们排除在首版之外，可以显著降低同步耦合与失败面。

## 数据格式

### 1. 明文同步包

应用在内存中先组装一个明文同步包：

```json
{
  "type": "accountSyncPlain_v1",
  "appVersion": "0.26.x",
  "exportedAt": 1770000000000,
  "settings": {},
  "mediaSource": {
    "connections": [],
    "credentials": {},
    "importRules": []
  }
}
```

这个明文包只在内存中使用，不直接上传，也不要求先落地成本地明文文件。

### 2. 加密同步文件

真正上传到 WebDAV 的内容是一个 JSON 外壳的加密信封：

```json
{
  "type": "accountSyncEncrypted_v1",
  "appVersion": "0.26.x",
  "exportedAt": 1770000000000,
  "cipher": {
    "algorithm": "AES-128-CBC",
    "kdf": "password+salt",
    "payloadType": "accountSyncPlain_v1",
    "salt": "base64",
    "iv": "base64",
    "ciphertext": "base64"
  }
}
```

设计原则：

1. 外层保持 JSON，便于后续做版本判断与调试；
2. 内层明文包整体加密，不拆字段级单独加密；
3. 未来如果加密方案升级，只需新增 `type / algorithm / kdf` 版本，不破坏旧文件识别。

## 加密方案

### 1. 加密方式

首版采用现有原生加密桥可直接支持的：

1. `AES/CBC/PKCS7Padding`

同步密码不直接作为 AES key 使用，而是先结合随机 `salt` 生成稳定长度的对称密钥，再配合随机 `iv` 完成加密。

首版密钥派生规则固定为：

1. 使用现有原生摘要能力计算 `SHA1(syncPassword + '\n' + salt)`；
2. 取结果前 16 字节作为 `AES-128-CBC` 的对称密钥；
3. 不在首版引入新的 `PBKDF2 / scrypt / argon2` 依赖。

这样可以在不新增密码学依赖的前提下，把加密规则固定成可复现、可测试、可兼容的单一路径。

### 2. 密码规则

1. 用户每次上传时手动输入同步密码；
2. 同步密码必须重复确认一次；
3. 两次输入不一致则禁止继续；
4. 同步密码不保存到本地设置、不进入同步账号配置、不进入媒体来源仓库。

### 3. 随机参数

每次上传都生成新的：

1. `salt`
2. `iv`

即使同步内容没有变化，只要重新上传，也会得到新的密文结果。

## 同步账号本地模型

同步账号建议单独存储为一个独立对象，不混入现有媒体来源仓库或普通 setting 主体：

```json
{
  "version": 1,
  "profile": {
    "displayName": "家里 NAS 同步",
    "serverUrl": "https://example.com/dav",
    "username": "user",
    "password": "password",
    "remoteDir": "/Apps/juMusicSync"
  },
  "validationKey": "string",
  "lastValidatedAt": 1770000000000,
  "lastUploadAt": 1770001000000,
  "lastUploadStatus": "success",
  "lastUploadMessage": ""
}
```

字段语义：

1. `profile`：同步账号配置本体；
2. `validationKey`：对当前可验证字段做摘要，用于判断“当前保存配置是否就是上次验证通过的那份”；
3. `lastValidatedAt`：最近成功验证时间；
4. `lastUploadAt`：最近成功上传时间；
5. `lastUploadStatus`：最近一次上传结果；
6. `lastUploadMessage`：最近一次失败或成功摘要。

同步账号配置仅用于“把同步包上传到哪里”，不属于首版账号同步包本体，因此不进入 `accountSyncPlain_v1`。

## WebDAV 交互规则

### 1. 测试连接

“测试连接”不只验证地址格式，还要尽量贴近真实上传路径：

1. 校验 `serverUrl / username / password / remoteDir` 必填；
2. 对 WebDAV 服务发起认证请求，确认凭据有效；
3. 检查 `remoteDir` 是否存在；
4. 若目录不存在，则尝试确认其父路径是否可访问；
5. 如果目录缺失但父路径可写，允许提示“首次上传时会自动创建”并视为可保存。

### 2. 创建远端目录

上传前需保证远端目录存在：

1. 对 `remoteDir` 按层级逐级尝试 `MKCOL`；
2. 已存在的目录视为成功；
3. 任一层目录在确认为不可创建时，上传直接失败并给出明确错误。

### 3. 远端文件路径

首版固定写入：

```text
<remoteDir>/jumusic-sync/account-sync.latest.json
```

优点：

1. 用户最好理解；
2. 首版无需管理历史版本；
3. 后续若要扩展 `latest + snapshots` 结构，不需要推翻当前路径语义。

### 4. 上传方法

上传动作统一采用：

1. `PUT`

上传内容为加密信封 JSON 字符串，不上传本地临时文件路径，也不依赖系统文件选择器。

## 用户操作流程

### 1. 首次配置流程

1. 进入“设置 -> 备份与恢复 -> 账号同步”；
2. 点击“配置 WebDAV”；
3. 输入显示名称、地址、用户名、密码、远端目录；
4. 点击“测试连接”；
5. 测试成功后点击“保存”；
6. 返回备份页并显示状态摘要。

### 2. 手动上传流程

1. 点击“上传同步”；
2. 如果当前配置未保存或未验证，先阻止继续并提示；
3. 弹出同步密码输入框；
4. 组装明文同步包；
5. 使用同步密码加密为信封 JSON；
6. 确保远端目录存在；
7. `PUT` 到固定文件路径；
8. 写入最近上传状态并提示结果。

## 错误处理

首版需要把错误归约成用户可理解的提示，不直接暴露冗长堆栈：

1. 未配置同步账号；
2. 当前配置尚未验证；
3. WebDAV 地址格式无效；
4. 认证失败；
5. 远端目录不可访问；
6. 远端目录创建失败；
7. 同步密码为空；
8. 两次同步密码不一致；
9. 加密失败；
10. 上传失败；
11. 网络异常或超时。

UI 表现：

1. 过程提示使用现有 `toast` 风格；
2. 最近失败摘要保留在“账号同步”区块，便于用户回看；
3. 成功上传后清理失败摘要并更新时间。

## 实现拆分

### 1. 账号同步数据模块

新增一个纯逻辑模块，负责：

1. 从当前设置与媒体来源仓库组装 `accountSyncPlain_v1`；
2. 明确筛除不应进入同步包的数据；
3. 生成同步账号的 `validationKey`；
4. 构建远端文件路径。

### 2. 加密模块

新增一个纯逻辑加密模块，负责：

1. 基于同步密码、`salt`、`iv` 生成密钥；
2. 调用现有原生 AES 能力加密整份明文包；
3. 生成 `accountSyncEncrypted_v1` 信封结构。

### 3. WebDAV 上传模块

新增一个轻量 WebDAV helper，职责为：

1. 复用现有请求头与 URL 拼装逻辑；
2. 提供目录检查 / `MKCOL` / `PUT` 能力；
3. 把网络与 HTTP 错误统一转成可读错误。

### 4. 备份页 UI

在现有 Backup 设置页中新增：

1. 账号同步区块；
2. WebDAV 配置弹窗；
3. 同步密码上传弹窗；
4. 最近状态摘要展示。

## 测试策略

1. 纯逻辑测试：
   1. 账号同步 payload 只包含 `settings + mediaSource(connections/credentials/importRules)`；
   2. 明确排除播放历史、统计缓存、sourceItems、snapshots；
   3. `validationKey` 在字段变化后会改变；
   4. 加密信封结构包含 `type / salt / iv / ciphertext`；
   5. 两次上传因随机 `salt / iv` 不应生成完全相同密文。
2. WebDAV 交互测试：
   1. 目录不存在时会逐级 `MKCOL`；
   2. 最终使用 `PUT` 上传固定路径；
   3. 认证失败与网络失败会转成明确错误。
3. UI 接线测试：
   1. “备份与恢复”页出现“账号同步”分组；
   2. 存在“配置 WebDAV”“上传同步”两个入口；
   3. 未验证配置时上传按钮会先被阻断；
   4. 上传时要求输入并确认同步密码。
4. 首版不做视觉截图测试。

## 后续扩展

如果首版上传能力稳定，后续可沿当前格式继续扩展：

1. 下载恢复；
2. 历史快照与 `latest` 指针；
3. 多设备最近上传信息比较；
4. 只同步部分设置或多 profile；
5. 把播放历史 / 年终统计作为单独可选同步包，而不是混进账号同步主包中。
