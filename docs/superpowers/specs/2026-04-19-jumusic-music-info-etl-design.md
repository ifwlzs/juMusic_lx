# juMusic 本地歌曲基础信息落库脚本设计

日期：2026-04-19  
状态：已确认，待用户审阅

## 概述

当前仓库还没有一个面向外部数仓的本地歌曲基础信息采集脚本。你的歌曲文件存放在 `Z:\Music`，目标是先用 Python 做一版可执行、可测试、可幂等重跑的基础信息落库能力，把本地音频文件的文件属性、标签信息、基础技术信息写入 SQL Server，供后续数仓层使用。

本设计首版只解决“把本地歌曲基础信息稳定采集到 ODS 层”这一件事，不扩展到复杂的增量删除、维表拆分、宽表派生、调度编排或下游 DWD/DWS 建模。

## 目标

1. 提供一个可直接执行的 Python 脚本，扫描 `Z:\Music` 下的音频文件。
2. 读取每首歌曲的文件基础信息、标签信息、音频技术信息。
3. 自动建表并将结果写入 SQL Server 表 `ods_jumusic_music_info`。
4. 支持重复运行时按 `file_path` 幂等更新，不重复插入同一路径记录。
5. 为每次运行生成 `batch_id`，便于追踪本批扫描结果。
6. 对解析失败或部分异常的文件保留记录，并写入错误摘要，避免整批任务中断。
7. 首版先以 `Z:\Music` 中部分歌曲做抽样测试，再扩大范围。

## 非目标

1. 不在首版实现中处理“本次没扫到的文件即判定删除”。
2. 不在首版实现中做复杂调度、任务编排或服务化部署。
3. 不拆分到多张 ODS 明细表、维表、异常表。
4. 不实现跨目录重命名识别、文件搬家识别、内容去重归并等高级能力。
5. 不在首版实现中把数据库凭据改造成独立配置中心或密钥托管方案。

## 用户输入与运行范围

### 输入范围

首版固定扫描根目录：

- `Z:\Music`

测试阶段按三步推进：

1. 抽样测试：选取 `Z:\Music` 下少量歌曲验证解析与入库；
2. 子目录测试：对单个子目录做完整扫描；
3. 全量测试：对整个 `Z:\Music` 执行全量扫描。

### 支持格式

首版默认支持以下常见音频扩展名：

- `.mp3`
- `.flac`
- `.m4a`
- `.aac`
- `.wav`
- `.ape`
- `.ogg`
- `.opus`

后续如遇特殊格式，再按实际测试结果补充。

## 方案选择

本次采用“**单表 ODS 宽表 + Python 直接解析元数据 + 按路径幂等 upsert**”方案。

### 选型理由

1. 落地最快，最适合当前“先把基础信息落下来”的目标；
2. 逻辑清晰，便于先验证目录扫描、中文路径、中文标签、SQL Server 连通性；
3. 只建一张主表即可满足首版数仓采集要求；
4. 通过预留 `batch_id`、`scan_status`、`scan_error` 等字段，为后续扩展扫描批次表或异常表留出空间。

## 目标表设计

### 表名

- `ods_jumusic_music_info`

命名遵循你要求的数仓规范：

- `层级_jumusic_表名`

### 主键与幂等策略

首版采用：

1. 物理主键：`id bigint identity(1,1)`；
2. 业务幂等键：`file_path`；
3. 重复运行时按 `file_path` 执行 upsert：
   - 已存在则更新；
   - 不存在则插入。

选择 `file_path` 作为首版幂等键的原因：

1. 当前测试范围是本地音乐目录，路径天然稳定；
2. 音频标签可能被用户手工修改，但路径在当前场景下更适合判定“同一首已采集文件”；
3. 首版先追求稳定采集与重跑幂等，后续如要引入内容哈希主键，可在 ODS 之上再扩展。

## 字段设计

### 主键与批次字段

- `id`：自增主键
- `batch_id`：本次扫描批次号
- `root_path`：扫描根目录
- `file_path`：文件完整路径
- `file_name`：文件名
- `file_ext`：扩展名

### 文件基础字段

- `file_size`：文件大小，单位字节
- `file_mtime`：文件最后修改时间
- `file_md5`：文件 MD5
- `is_readable`：文件是否可正常读取

### 标签字段

- `title`
- `artist`
- `album`
- `album_artist`
- `track_no`
- `disc_no`
- `genre`
- `year`

### 音频技术字段

- `duration_sec`
- `bitrate`
- `sample_rate`
- `channels`

### ETL / 状态字段

- `scan_status`：`SUCCESS` / `FAILED`
- `scan_error`：错误摘要
- `etl_created_at`
- `etl_updated_at`

## SQL Server 字段类型建议

| `file_ext` | `varchar(20)` | 扩展名 |

| `file_mtime` | `datetime2` | 最后修改时间 |

## 索引设计

首版至少创建以下索引：

1. `file_path` 唯一索引；
2. `batch_id` 普通索引。

目的：

1. 用 `file_path` 保证 upsert 幂等；
2. 用 `batch_id` 快速查看某次扫描的结果与异常记录。

## Python 脚本设计

### 目标位置

脚本放在：

- `scripts/music_etl/load_music_info.py`

首版先保持单文件实现，降低落地成本；若后续复杂度增加，再拆成模块。

### 内部函数边界

即使首版是单文件，也按以下职责拆分：

1. `iter_music_files(root_path)`
   - 递归扫描音频文件
   - 过滤支持的扩展名
2. `extract_file_info(file_path)`
   - 读取文件基础属性
   - 计算文件 MD5
3. `extract_audio_metadata(file_path)`
   - 读取标签信息与音频技术信息
4. `ensure_table(conn)`
   - 检查并创建目标表与必要索引
5. `upsert_music_rows(conn, rows)`
   - 批量执行更新与插入
6. `main()`
   - 串联参数、扫描、建表、入库、统计输出

## 依赖选择

### 元数据解析

首版使用 `mutagen` 读取音频元数据。

原因：

1. 对 `mp3 / flac / m4a / ogg` 等常见格式支持成熟；
2. 纯 Python 依赖，安装和脚本落地都比较轻；
3. 能同时提供标签信息与部分技术信息。

### SQL Server 连接

首版使用 Python 的 SQL Server 驱动进行直连写入。实现层面优先选择成熟稳定、参数化支持较好的方案，例如：

- `pymssql` 或 `pyodbc`

最终以本机测试安装最顺利、连接最稳定的驱动为准，但脚本接口不依赖某个数据库 ORM。

## 元数据解析策略

解析原则是“**能读多少读多少，不因单文件异常中断整批**”。

### 规则

1. 文件存在且可访问时，尽量保留一条记录；
2. 能解析到的字段正常写入；
3. 某个标签缺失时，对应字段记空，不视为失败；
4. 文件不可读、解析器异常、标签损坏等情况：
   - `scan_status = 'FAILED'`
   - `scan_error` 记录异常摘要；
   - 仍尽量保留文件基础信息；
5. 文件存在、元数据部分为空但整体可读取时：
   - `scan_status = 'SUCCESS'`
   - `scan_error = null`

### 标签兼容策略

由于不同音频格式的标签字段命名不同，脚本需要做统一映射：

1. `title` 统一映射到标题字段；
2. `artist` 统一映射到歌手字段；
3. `album` 统一映射到专辑字段；
4. `album_artist` 统一映射到专辑艺术家字段；
5. `track_no`、`disc_no` 对 `1/12` 这类值做整数提取；
6. `year` 保留原始日期/年份主值，不在首版强行规范成完整日期维度。

## 扫描与入库策略

### 扫描模式

首版执行**全量扫描**，但不做删除判断：

1. 每次运行遍历当前扫描范围内的文件；
2. 为本次运行生成新的 `batch_id`；
3. 对扫描到的文件执行 upsert；
4. 不因为某条旧记录本次未被扫描到，就将其删除或失效。

这样可以避免在首版就引入高风险的删除策略。

### Upsert 策略

首版采用“**先更新、后插入**”方式，而不是直接使用 `MERGE`。

原因：

1. 语义更直接；
2. 对参数化 SQL 和错误定位更友好；
3. 能减少部分 SQL Server 环境下 `MERGE` 的兼容性与行为坑。

执行逻辑：

1. 按 `file_path` 尝试更新现有记录；
2. 若不存在，再插入新记录；
3. 更新时刷新：
   - 元数据字段
   - 技术字段
   - `batch_id`
   - `scan_status`
   - `scan_error`
   - `etl_updated_at`
4. 插入时同时写入：
   - `etl_created_at`
   - `etl_updated_at`

## 日志与统计输出

脚本控制台输出以简洁统计为主，避免刷屏：

1. 扫描文件总数；
2. 成功解析数；
3. 失败解析数；
4. 更新条数；
5. 插入条数；
6. 总耗时。

必要时附带前若干条失败文件摘要，便于快速定位异常格式或坏文件。

## 测试策略

### 阶段 1：抽样测试

从 `Z:\Music` 中选取 10~30 首歌曲进行测试，优先覆盖：

1. 中文路径；
2. 中文标题/歌手/专辑；
3. 不同格式，如 `mp3`、`flac`、`m4a`；
4. 可能缺少标签或标签不规范的文件。

验证点：

1. 表自动创建成功；
2. 数据库连接成功；
3. 中文路径和中文元数据入库正常；
4. `file_path` 唯一键生效；
5. 失败记录能正确写入 `scan_error`。

### 阶段 2：子目录测试

对 `Z:\Music` 下一个完整子目录执行扫描。

验证点：

1. 性能是否可接受；
2. 重复运行是否正确走更新逻辑；
3. 异常文件是否会拖垮整批。

### 阶段 3：全量测试

在前两阶段稳定后，再扫描完整 `Z:\Music`。

验证点：

1. 批量扫描稳定性；
2. 大量中文文件路径与标签兼容性；
3. 长时间运行下的数据库写入稳定性。

## 错误处理

1. 单文件处理失败不影响整体任务继续执行；
2. 数据库连接失败时任务直接失败并输出明确错误；
3. 建表失败时不继续执行扫描入库；
4. 批量 upsert 过程中出现单条异常时，应至少记录失败文件路径与异常信息，便于后续补扫；
5. 对无法获取 MD5 或某些技术信息的文件，不默认判定为整条失败，除非文件整体不可读。

## 安全与敏感信息处理

1. 数据库连接信息属于敏感配置，不写入文档示例、不打印到日志中；
2. 控制台日志避免输出完整连接串与账号密码；
3. 如需调试数据库连接，仅输出主机、端口、库名等非敏感摘要。

## 验收标准

1. 能在本地直接执行 Python 脚本扫描 `Z:\Music` 中的测试歌曲；
2. 脚本能自动创建 `ods_jumusic_music_info`；
3. 每首测试歌曲至少能落入一条记录；
4. 重复运行不会因相同 `file_path` 产生重复数据；
5. 中文路径、中文标签能正确写入 SQL Server；
6. 元数据异常文件不会导致整批中断；
7. 失败文件可通过 `scan_status` 和 `scan_error` 定位；
8. 脚本输出清晰的成功/失败/插入/更新统计。

## 后续扩展方向

首版完成后，可按需要逐步扩展：

1. 增加 `ods_jumusic_scan_batch` 扫描批次表；
2. 支持仅扫描指定子目录或限制文件数量，方便调试；
3. 引入增量扫描条件，如按文件修改时间跳过未变化文件；
4. 补充软删除或失效标记逻辑；
5. 在 ODS 之上继续沉淀 DWD 层标准化歌曲主题表。
