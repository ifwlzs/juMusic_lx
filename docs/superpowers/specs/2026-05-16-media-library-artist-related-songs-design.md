# 我的列表歌手相关歌曲入口设计规范

日期：2026-05-16  
状态：待审阅（待按此规格进入实现计划）

## 1. 目标与范围

本规范用于指导 juMusic 在「我的列表」场景下补齐“点击歌手查看相关歌曲”的最小闭环能力，优先解决以下问题：

1. 用户在 `MusicDetailModal` 中看到歌手信息，但当前只是静态文本，无法继续查看同歌手歌曲；
2. `WebDAV / SMB / OneDrive / 本地` 歌曲虽然已经可以打开应用内详情弹窗，但用户仍缺少从详情直接回看同歌手歌曲的入口；
3. 当前列表已经有搜索结果浮层能力，但它只服务于手动输入关键字，未复用到“基于当前歌曲歌手查看相关歌曲”的场景。

本版目标是提供一个**仅限当前列表范围**、**仅在歌曲详情弹窗里触发**、**点击后可直接定位到目标歌曲**的歌手相关歌曲入口。

本版不包含：

- 独立的歌手详情页或歌手歌曲页；
- 列表行内歌手名点击；
- 在线音源外链详情逻辑重构；
- 跨列表、全媒体库或全局歌曲搜索；
- 联名歌手拆分、推荐歌手或批量播放能力。

---

## 2. 当前问题与根因

### 2.1 当前现象

在「我的列表」中：

1. 本地 / 媒体库歌曲打开 `MusicDetailModal` 后，可以看到歌手字段；
2. 歌手字段当前只是普通文本，无法继续操作；
3. 用户若想查看同歌手歌曲，只能退出详情后再手动使用列表搜索，链路长且不直观；
4. 对 `WebDAV` 等媒体库列表而言，用户甚至连“先复制歌手名再搜”这条兜底路径都不顺手。

### 2.2 根因

#### 根因 A：详情弹窗当前只有只读展示，没有字段级交互

`src/components/MusicDetailModal/index.tsx` 目前仅负责展示歌曲摘要、详情分组与复制动作，歌手字段在摘要区和基本信息区都只是 `Text`，没有可点击回调。

#### 根因 B：现有结果浮层只支持关键字搜索模式

`src/screens/Home/Views/Mylist/MusicList/ListMusicSearch.tsx` 当前只有：

- `search(keyword, height)`
- `hide()`

其内部刷新逻辑也只围绕 `searchListMusic(...)` 展开，因此无法直接承接“按当前歌曲歌手展示结果”的需求。

#### 根因 C：当前列表缺少“按歌手筛同列表歌曲”的纯函数能力

`src/screens/Home/Views/Mylist/MusicList/listAction.ts` 目前只有面向关键字模糊搜索的 `searchListMusic(...)`，没有面向“完整歌手字符串匹配”的纯函数入口。

## 3. 设计原则

### 3.1 入口只加一个，放在用户已经打开的详情弹窗里

本版不新增列表行入口，也不新增页面路由，只在 `MusicDetailModal` 中让歌手可点。原因：

1. 这是当前用户已经明确确认的最小入口；
2. 改动集中，风险小；
3. 不会打断现有列表单击播放、长按多选、更多菜单等已有操作。

### 3.2 结果层复用现有列表搜索浮层，不重复造 UI

本版直接复用 `ListMusicSearch` / `SearchTipList` 的浮层与列表渲染能力，只新增一种“歌手相关歌曲模式”，不再单独新建歌手弹窗或二级页面。

### 3.3 查询范围严格限制在当前列表

用户点击歌手后，只在**当前激活列表**中查找相关歌曲，不跨到其他自建列表、搜索页、媒体库总表或在线音源结果页。

### 3.4 匹配规则以当前详情显示的完整歌手字符串为准

本版不拆分联名歌手，也不做模糊推荐。用户点击的就是当前详情展示出来的完整歌手字符串，结果也按这整个字符串去匹配。

### 3.5 结果顺序优先保持当前列表顺序

歌手相关歌曲不是关键字搜索排序问题，因此本版结果直接保留当前列表里的原始顺序，不再做相似度排序。这样用户看到结果后，更容易理解它们在列表中的相对位置。

---

## 4. 入口与行为范围

### 4.1 本版生效入口

入口位置：

- `我的列表` → `歌曲详情弹窗` → `歌手`

### 4.2 本版覆盖的歌曲类型

由于 `MusicDetailModal` 当前只用于应用内详情链路，本版入口实际覆盖以下歌曲：

1. 本地歌曲；
2. 媒体库本地歌曲；
3. WebDAV 歌曲；
4. SMB 歌曲；
5. OneDrive 歌曲；
6. 其他后续会走 `MusicDetailModal` 的内部详情歌曲。

在线音源歌曲仍维持原有外链详情逻辑，因此本版不覆盖它们。

### 4.3 行为矩阵

| 场景 | 行为 |
| --- | --- |
| 详情弹窗中歌手有值 | 显示为可点击文本，点击后展示当前列表内该歌手相关歌曲 |
| 详情弹窗中歌手为空或仅空白 | 保持普通只读文本，不提供点击态 |
| 结果列表中点击某首歌 | 沿用现有 `onScrollToInfo(item)`，滚动定位到该歌曲并关闭结果浮层 |
| 点击结果浮层空白区域 | 关闭结果浮层 |
| 当前列表变化后结果仍有命中 | 用同一歌手条件重新计算并刷新结果 |
| 当前列表变化后结果为空 | 收起结果浮层，避免展示空壳列表 |

---

## 5. 匹配规则

### 5.1 基本规则

本版新增一个面向当前列表的纯函数，例如：

```ts
findArtistRelatedSongsInList(list, artist)
```

其匹配规则为：

1. 先对点击得到的歌手字符串做首尾空白裁剪；
2. 再对列表中每首歌的 `musicInfo.singer` 做同样裁剪；
3. 使用**完整字符串全等匹配**筛选结果；
4. 不拆分 `、 ; ； & / , ， |` 等联名分隔符；
5. 不按歌名、专辑、拼音或模糊关键字扩展结果。

### 5.2 结果包含当前歌曲

结果集合应包含当前点击详情的这首歌曲本身，只要它的 `singer` 字段与点击值一致即可。

### 5.3 结果顺序

返回结果时保持原列表顺序，不做额外排序。

### 5.4 空结果兜底

正常情况下，当前歌曲本身会命中，因此很少出现空结果。

若因列表切换、数据刷新或异常脏数据导致结果为空，则：

1. 不展示空白结果浮层；
2. 给出一次轻量 toast，提示“当前列表未找到该歌手相关歌曲”；
3. 不影响当前列表与详情页之外的其他状态。

## 6. 交互流程

### 6.1 主流程

1. 用户在 `MusicDetailModal` 中点击歌手；
2. 弹窗内部先关闭详情弹窗，避免与结果浮层叠层；
3. 页面层收到 `artist + musicInfo` 回调；
4. 页面层调用 `ListMusicSearch` 的“歌手相关歌曲模式”；
5. `ListMusicSearch` 读取当前激活列表，按完整歌手字符串过滤出相关歌曲；
6. 结果浮层沿用现有样式展示歌曲列表；
7. 用户点击任一结果项后，继续走现有 `handleScrollToInfo(item)`，滚动定位并关闭结果浮层。

### 6.2 浮层展示约定

本版直接复用现有结果浮层：

- 不新增歌手详情页；
- 不新增浮层标题；
- 不新增播放全部、批量选择、二次筛选等额外动作；
- 仅复用其“展示结果列表 + 点击后回滚定位”的能力。

### 6.3 列表更新行为

`ListMusicSearch` 当前已经订阅：

- `mylistToggled`
- `myListMusicUpdate`

本版应继续复用这一机制：

1. 当结果浮层处于“歌手相关歌曲模式”时，列表变化后按同一歌手条件重新计算；
2. 若更新后仍有结果，则刷新浮层内容；
3. 若更新后无结果，则直接收起浮层。

这样可以保证它与现有关键字搜索浮层在“跟随当前列表变化”这一点上保持一致。

## 7. UI 与可用性约定

### 7.1 歌手点击态

详情弹窗中的歌手字段需要有明确的可点击提示，建议至少满足以下几点：

1. 使用与普通只读字段不同的视觉样式，例如主题主色文字或轻微下划线提示；
2. 点击时保留现有按钮按压反馈；
3. 歌手为空时不展示点击态，避免误导。

### 7.2 弹窗内两个歌手展示位保持一致

当前 `MusicDetailModal` 同时存在：

- 顶部摘要区歌手；
- 基本信息分组里的歌手字段。

本版要求这两个展示位的行为保持一致：

- 要么都可点；
- 要么都不点；

避免同一个“歌手”在同一弹窗里出现一处可点、一处不可点的割裂体验。

### 7.3 结果项展示

结果浮层继续沿用当前 `ListMusicSearch` 的列表项结构：

- 主标题：歌名；
- 副标题：歌手 + 专辑；
- 右侧：来源。

本版不为歌手相关歌曲模式新增定制化结果卡片。

## 8. 推荐实现落点

### 8.1 `src/components/MusicDetailModal/index.tsx`

建议新增一个可选回调属性，例如：

```ts
onPressArtist?: (payload: { artist: string, musicInfo: LX.Music.MusicInfo }) => void
```

约定如下：

1. `show(musicInfo)` 的 imperative API 保持不变；
2. 由弹窗内部负责先关闭自己，再触发 `onPressArtist`；
3. 当歌手为空时，不触发回调；
4. UI 层只负责点击态与回调派发，不承担列表过滤逻辑。

### 8.2 `src/components/MusicDetailModal/buildDetailSections.ts`

本版建议**维持当前纯只读详情模型契约不变**，不为了这一期单一交互强行给 section item 注入额外 action 元数据。

原因：

1. 当前点击能力只针对“歌手”这一处字段；
2. 由 `MusicDetailModal` UI 层结合 `musicInfo.singer` 与 `onPressArtist` 做判断即可；
3. 可避免把展示纯函数扩大成过度通用的交互描述协议。

若后续同类可点击字段继续增加，再考虑在详情模型层抽象交互元数据。

### 8.3 `src/screens/Home/Views/Mylist/MusicList/ListMusicSearch.tsx`

建议扩展为双模式：

1. 关键字搜索模式：保留现有 `search(keyword, height)`；
2. 歌手相关歌曲模式：新增例如 `showArtistRelatedSongs(artist, height)`。

内部建议改为统一的查询上下文，例如：

- `type: 'keyword'`
- `type: 'artist'`

然后复用一套“读取当前列表 → 计算结果 → 更新 SearchTipList”的刷新流程，避免关键字模式与歌手模式出现两套分叉实现。

### 8.4 `src/screens/Home/Views/Mylist/MusicList/listAction.ts`

建议新增一个纯函数，例如：

```ts
findArtistRelatedSongsInList(list, artist)
```

职责仅限：

1. 标准化歌手字符串；
2. 在当前列表中过滤命中项；
3. 保持原顺序返回结果。

它不负责 UI、toast、滚动或弹窗控制，便于后续单测覆盖。

### 8.5 `src/screens/Home/Views/Mylist/MusicList/index.tsx`

页面层负责把两端接起来：

1. 给 `MusicDetailModal` 传入 `onPressArtist`；
2. 在回调里调用 `listMusicSearchRef.current?.showArtistRelatedSongs(...)`；
3. 继续复用现有 `handleScrollToInfo(info)`，不新增第二套滚动定位逻辑。

### 8.6 i18n 文案

若实现采用空结果 toast，则需补齐至少以下文案键：

- `music_detail_artist_related_empty`

并同步维护：

- `src/lang/zh-cn.json`
- `src/lang/zh-tw.json`
- `src/lang/en-us.json`

## 9. 实现边界

### 9.1 本版明确要做

1. 只在 `MusicDetailModal` 里让歌手可点击；
2. 点击后仅展示当前列表内完整歌手字符串匹配到的歌曲；
3. 结果浮层复用现有 `ListMusicSearch` / `SearchTipList`；
4. 结果项点击后继续滚动定位到歌曲；
5. 为空结果提供轻量反馈（若实现采用 toast 方案）。

### 9.2 本版明确不做

1. 不新增歌手全屏页；
2. 不在列表行中让歌手可点；
3. 不做跨列表、跨媒体库、跨来源搜索；
4. 不拆分联名歌手；
5. 不增加“播放全部”“收藏歌手”“查看专辑”等派生功能；
6. 不重写在线音源的歌曲详情入口。

## 10. 测试策略

### 10.1 需要覆盖的核心点

1. 详情弹窗歌手有值时，可触发歌手相关歌曲回调；
2. 歌手为空时，不触发点击回调，也不展示误导性点击态；
3. `findArtistRelatedSongsInList(...)` 会：
   - 只按完整歌手字符串匹配；
   - 忽略首尾空白差异；
   - 不拆分联名歌手；
   - 保持原列表顺序；
4. `ListMusicSearch` 在歌手模式下可以展示结果，并继续复用点击后滚动定位逻辑；
5. 当前列表更新后，歌手模式会按相同条件刷新结果；
6. 当刷新后结果为空时，会收起浮层并给出预期反馈（若实现采用 toast 方案）。

### 10.2 测试层级建议

优先按现有项目测试风格补以下两类：

- 纯函数测试：覆盖 `findArtistRelatedSongsInList(...)` 的匹配规则；
- 组件 / 行为测试：覆盖 `MusicDetailModal` 歌手点击分流与 `ListMusicSearch` 歌手模式的结果更新。

若当前测试基础不足，可先保证纯函数与关键分流测试，再视现有测试设施补更细的 UI 行为断言。

## 11. 后续演进方向（非本版范围）

以下内容不属于本版实现范围，但本版设计不应阻断它们：

1. 从歌手结果浮层进一步升级为独立歌手页；
2. 在列表行、副标题或播放页中增加更多歌手入口；
3. 支持联名歌手拆分与单歌手聚合；
4. 扩展为当前媒体库或全部列表范围的歌手歌曲浏览；
5. 增加歌手维度的排序、播放全部、批量操作。
