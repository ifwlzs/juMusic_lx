# 媒体来源默认增量同步与手动全量校验设计

日期：2026-04-09  
状态：已确认，待用户审阅

## 概述

当前媒体来源更新流程在导入规则发生变更时，仍然容易退回“把当前规则全部已选目录和已选歌曲重新扫描一遍”的重路径。对用户来说，这会带来两个明显问题：

1. 明明只是给现有规则补加一个目录或一首歌，却要等待整个旧范围重新扫描；
2. 日常点一次“更新”，会把大量没有变化的旧目录重新走一轮扫描，体感很慢。

本设计把同步流程拆成两档：

1. **默认更新**：走增量同步，只优先补“新加范围”“近期新增歌曲”“近期更新歌曲”；
2. **手动全量校验 / 重新扫描**：只在用户显式触发时，才做全量核对与删歌判断。

这样可以把“日常快速更新”和“严格一致性校验”明确分开，优先保证新歌尽快出现，同时避免因为删歌判断而把每次更新都拖成近似全量扫描。

## 目标

1. 编辑导入规则后，如果只是新增目录或新增歌曲，不再对整个既有范围做完整重扫。
2. 默认“更新”优先同步：
   1. 新增导入范围中的歌曲；
   2. 近期新增歌曲；
   3. 近期更新过的歌曲。
3. 默认“更新”不处理源端删歌，避免为了判断删除而退回全量扫描。
4. 新增一个显式的“全量校验 / 重新扫描”入口，专门负责完整核对与删歌处理。
5. 用户手动从规则里移除目录或歌曲时，这部分范围要立即从规则结果中移除，不等全量校验。
6. 本地、SMB、WebDAV、OneDrive 等来源尽量统一到“轻量枚举 + 选择性补 metadata”的同步模式。

## 非目标

1. 不新增后台自动全量校验任务。
2. 不在本次实现中做文件搬家识别、重命名智能识别或 path rename 补偿。
3. 不在默认更新里处理源端删除传播。
4. 不改动已有的系统歌单结构与列表投影模型。
5. 不把远端 provider 的流式同步二期框架重写成另一套独立模型。

## 当前问题

### 1. 规则变更后仍然容易整轮重扫

`updateImportRule()` 最终仍会调用 `syncImportRule()`。对本地 provider 来说，`syncImportRule()` 当前直接走 `scanImportSelection()`，其行为是：

1. 把当前规则的全部已选目录递归扫描；
2. 把当前规则的全部已选歌曲逐一检查；
3. 再统一生成下一版快照。

这意味着：

1. 只新增一首歌，也会把旧目录重新走一遍；
2. 规则范围越大，编辑一次规则的成本越高。

### 2. 默认更新把“补新歌”和“删旧歌判断”绑在了一起

现有逻辑里，完整扫描成功后就会用新结果直接和旧快照对比并处理删除。这在“全量校验”场景是合理的，但不适合作为日常默认更新策略，因为：

1. 删歌判断天然依赖足够完整的范围扫描；
2. 只要要保证删歌判断正确，就容易把默认更新拖回重路径；
3. 用户体感上真正更重要的是“新歌先出现”。

### 3. 用户显式缩小规则范围和源端删歌是两种不同语义

这两类变化当前都容易落到“重新扫描后再整体对比”的处理方式里，但语义并不相同：

1. **用户移除目录 / 歌曲**：这是规则显式变更，应该立即从规则结果里消失；
2. **源端文件已删除**：这属于外部变化，只有在完整校验后才能安全删除。

这两种情况需要在产品和实现上明确拆开。

## 用户可见行为

## 1. 默认“更新”

规则卡片和账号卡片上的“更新”按钮改为**快速更新**语义：

1. 优先同步新加范围；
2. 再补近期新增歌曲和近期更新歌曲；
3. 不处理源端删歌；
4. 更新失败时保留旧结果。

用户日常使用这个按钮时，核心体验是“尽快把新内容补上”，而不是“做一次完整一致性审计”。

## 2. 新增“全量校验 / 重新扫描”

规则级和账号级都新增一个显式入口：

1. `全量校验`
2. 或文案为 `重新扫描`

该入口的语义是：

1. 对当前规则或当前账号下规则做完整扫描；
2. 根据完整扫描结果判断删歌；
3. 重建完整快照基线；
4. 作为“严格对齐源端状态”的手动动作。

## 3. 编辑规则后的行为

### 新增目录 / 新增歌曲

保存规则后默认走增量同步：

1. 新增目录先扫描新增目录自身；
2. 新增歌曲先检查新增歌曲自身；
3. 新内容优先进入系统歌单和聚合结果；
4. 不对整个旧范围做完整 metadata 重扫。

### 移除目录 / 移除歌曲

保存规则后立即生效：

1. 被规则移除的范围不再属于当前规则；
2. 若这些歌曲未被同账号其他规则覆盖，则立即从该规则投影结果中移除；
3. 这不属于“源端删歌”，因此不需要等待全量校验。

## 同步架构

## 1. 双模式同步

同步流程明确分为两种模式：

### `incremental`

默认更新模式，用于：

1. 编辑规则后的自动补同步；
2. 用户平时点击“更新”；
3. 自动同步触发。

特点：

1. 只处理新增范围、近期新增、近期更新；
2. 不做源端删歌判断；
3. 更偏向“尽快让用户看到新歌”。

### `full_validation`

手动全量校验模式，用于：

1. 用户主动点击“全量校验 / 重新扫描”；
2. 账号级统一重建完整基线。

特点：

1. 做完整范围枚举；
2. 允许删歌；
3. 刷新完整快照与统计基线。

## 2. 规则差异集

在 `updateImportRule()` 开始时，先比较 `previousRule` 和 `rule`，拆出三类集合：

1. `addedSelections`
   - 新增目录
   - 新增歌曲
2. `removedSelections`
   - 被规则移除的目录
   - 被规则移除的歌曲
3. `unchangedSelections`
   - 规则前后都存在的范围

后续处理顺序固定为：

1. 先应用 `removedSelections`；
2. 再优先同步 `addedSelections`；
3. 最后对 `unchangedSelections` 做轻量枚举与候选筛选。

## 3. removedSelections 的处理

`removedSelections` 是用户显式缩小规则范围，不依赖源端实际文件状态。

处理原则：

1. 基于当前规则快照中的 `items` 判断哪些歌曲只属于被移除范围；
2. 如果这些歌曲没有被同账号其他规则覆盖，则立即从当前规则结果中移除；
3. 若仍被同账号其他规则覆盖，则只从当前规则结果移除，但保留在账号聚合结果中；
4. 这一阶段不调用“删歌传播到自建歌单”的源端删除逻辑。

这样可以保证“用户手动缩小规则范围”立刻可见，而不会被误当成“等待下一次完整扫描再判断”的事情。

## 4. addedSelections 的优先同步

`addedSelections` 总是优先处理：

1. 新增目录：只对新增目录做候选枚举；
2. 新增歌曲：只检查新增歌曲；
3. 对新增范围内的候选项直接进入 metadata 补全过程；
4. 新歌尽快写入 source items、聚合结果和系统歌单。

这一步的目标不是“校验整个规则是否完全一致”，而是让新增范围尽快完成首轮可见。

## 5. unchangedSelections 的轻量枚举

对未变化的既有范围，不再直接做完整 `scanSelection()`，而是先做**轻量枚举**：

1. 只拿路径、文件名、文件大小、`modifiedTime`、`versionToken` 等轻量信息；
2. 不在这一阶段统一读取 metadata；
3. 只用来判断哪些歌曲需要补同步。

### 轻量枚举输出

每个 selection 需要生成一份统计：

1. `itemCount`
2. `latestModifiedTime`
3. `selectionKey`

以及一批候选项：

1. `sourceStableKey`
2. `pathOrUri`
3. `fileSize`
4. `modifiedTime`
5. `versionToken`

## 6. 增量候选筛选规则

轻量枚举完成后，只对以下候选进入 metadata 补全过程：

1. 上次快照中不存在的候选；
2. `versionToken` 发生变化的候选；
3. `modifiedTime > lastIncrementalSyncAt` 的候选；
4. 所属 selection 的 `itemCount` 与上次不一致，且候选需要入库或更新的。

也就是说，默认更新不会对“没有变化的旧歌”重新读 metadata。

## 7. 全量校验流程

当模式为 `full_validation` 时：

1. 对当前规则范围做完整扫描；
2. 生成完整结果集；
3. 用完整结果和当前快照对比；
4. 允许判定源端已删除歌曲；
5. 删除系统歌单和自建歌单中的相应引用；
6. 更新完整快照基线与 selection 统计基线。

## 数据模型

## 1. 扩展 import snapshot

继续沿用现有 `importSnapshot` 存储，但扩展字段：

1. `lastIncrementalSyncAt`
2. `lastFullValidationAt`
3. `selectionStats`
4. `pendingFullValidation`

建议结构：

```ts
{
  ruleId: string
  scannedAt: number | null
  items: SourceItem[]
  isComplete?: boolean
  lastIncrementalSyncAt?: number | null
  lastFullValidationAt?: number | null
  pendingFullValidation?: boolean
  selectionStats?: Array<{
    selectionKey: string
    kind: 'directory' | 'track'
    pathOrUri: string
    itemCount: number
    latestModifiedTime: number
    capturedAt: number
  }>
}
```

字段含义：

1. `items`：当前规则投影到列表的最新结果，可包含尚未经过最近一次全量校验的旧项；
2. `lastIncrementalSyncAt`：默认更新最近一次完成时间；
3. `lastFullValidationAt`：最近一次完整校验时间；
4. `selectionStats`：selection 级别的数量与最新修改时间基线；
5. `pendingFullValidation`：提示当前规则建议做一次全量校验。

## 2. selectionKey

`selectionStats` 使用稳定的 `selectionKey`：

1. 目录：`directory::<normalizedPath>`
2. 单曲：`track::<normalizedPath>`

这样可以在规则编辑前后稳定比较同一个 selection 的统计结果。

## Provider 合约调整

## 1. local provider

本地 provider 目前只有 `scanSelection()` 重路径，需要补齐：

1. `enumerateSelection(connection, selection)`
   - 只返回候选，不读 metadata；
2. `hydrateCandidate(connection, candidate)`
   - 只对选中的候选补 metadata。

这样本地来源才能和 SMB / WebDAV / OneDrive 一样，进入“轻量枚举 + 选择性补 metadata”的统一流程。

## 2. SMB / WebDAV / OneDrive

这些 provider 已经基本具备轻量枚举能力，本次只需要：

1. 在协调层优先使用 `enumerateSelection()` 或 `streamEnumerateSelection()`；
2. 仅对筛选出来的增量候选执行 `hydrateCandidate()`；
3. 保留现有完整扫描能力给 `full_validation` 使用。

## UI 与状态反馈

## 1. 规则卡片 / 账号卡片

需要把当前单一“更新”语义拆成两种按钮或二级操作：

1. `更新`
2. `全量校验 / 重新扫描`

## 2. 状态文案

建议同步状态摘要区分两类模式：

1. `最近快速更新`
2. `最近全量校验`

若不想新增两行字段，至少要在摘要中标出：

1. 本次是 `快速更新`
2. 还是 `全量校验`

## 3. pendingFullValidation 提示

当出现以下情况之一时，规则卡片可提示“建议全量校验”：

1. selection 数量变化过多；
2. 增量同步连续多轮后仍积累明显差异；
3. provider 返回不完整或统计异常。

这只是提示，不自动触发删歌。

## 错误处理

1. 默认增量更新失败时，不删除现有结果。
2. `addedSelections` 同步失败时，保留旧快照，仅记录失败状态。
3. 轻量枚举失败时，不执行删歌，也不清空规则结果。
4. 全量校验失败时，保留当前快照，不进入删除阶段。
5. selection 统计缺失时，默认退回“该 selection 全量补同步一次”，但仍不处理源端删歌。

## 验收标准

1. 给已有规则新增一个目录时，不再对整个旧范围做完整 metadata 重扫。
2. 给已有规则新增一首歌时，这首歌可以优先进入系统歌单。
3. 日常点击“更新”时，近期新增歌曲和近期更新歌曲能被补进来。
4. 默认“更新”不会因为做删歌判断而拖成整轮重扫。
5. 用户从规则里移除目录或歌曲时，这部分范围会立即从规则结果里消失。
6. 只有手动点击“全量校验 / 重新扫描”时，才处理源端删歌。
7. 增量同步失败时，旧列表结果保持不变。

## 自动化测试重点

1. 新增 selection 时只处理新增范围，不重扫整个旧范围。
2. `removedSelections` 会立即从规则结果中移除，但不会走源端删歌传播。
3. `enumerateSelection()` 只返回候选，不读 metadata。
4. `hydrateCandidate()` 只对筛出来的增量候选执行。
5. 数量一致但 `modifiedTime > lastIncrementalSyncAt` 的歌曲仍会被补同步。
6. 默认增量更新不会删除旧歌。
7. 全量校验会处理源端删除并更新快照基线。
