# 播放页配色与媒体来源歌单排序设计

日期：2026-04-06  
状态：已确认，待用户复核后进入实现规划

## 概述

本次设计只处理三个已经确认的产品问题：

1. 播放页在动态背景下的前景信息不稳定，封面颜色一变化，歌名、歌手、时间、图标和按钮就会丢对比度。
2. NAS / WebDAV / SMB 导入生成的只读歌单，在“我的列表”里的字体观感偏灰，与普通歌单不一致。
3. 媒体来源歌单当前不能使用排序能力，也没有记住上次排序方式的机制。

本次不重做全局主题系统，也不引入新的导航入口，只对播放页和媒体来源歌单的展示层与排序规则做收敛。

## 目标

1. 让播放页在有封面、无封面、深色封面、浅色封面几种场景下都保持稳定可读。
2. 让播放页前景的文本和图标形成统一视觉语言，不再混用一批灰色辅助色。
3. 让 NAS / WebDAV / SMB 生成歌单在“我的列表”里的字体层级与普通歌单一致，不因为只读属性变淡灰。
4. 为媒体来源歌单增加可用的排序入口，并让每个歌单记住自己的排序方式。
5. 新增 `更新时间` 和 `文件名` 两个排序字段。

## 非目标

1. 不重做整个 App 的主题 token 体系。
2. 不引入基于图片像素采样的亮度识别算法。
3. 不改变普通可编辑歌单当前“真实重排列表顺序”的行为。
4. 不把媒体来源歌单变成可编辑歌单，也不把排序结果回写到源端。

## 当前问题定位

### 播放页

当前播放页虽然已经有 `playDetailEmby` 背景预设，但前景层仍混用多种灰色 token：

1. 歌手名、时间、更多按钮、头部按钮等仍使用 `c-font-label`、`c-500`、`c-550` 一类辅助色。
2. 动态背景越强，灰色前景越容易和背景糊在一起。
3. 仅靠“加深模糊”不能解决问题，因为根因是前景对比度不足，而不是背景不够糊。

### 我的列表 / NAS 歌单

当前媒体来源歌单的“只读”逻辑虽然主要体现在菜单禁用上，但媒体来源歌单相关列表项依然混入了多处偏淡辅助色，导致视觉上看起来像“灰掉了”：

1. 序号、副信息、时长使用偏浅灰 token。
2. 不可用歌曲额外叠加较重透明度，会进一步放大“灰”的观感。
3. 普通歌单与媒体来源歌单没有明确共享同一套正文颜色策略。

### 排序

当前项目已有歌单排序弹窗，但它服务的是普通可编辑歌单：

1. 排序入口对只读媒体来源歌单是禁用的。
2. 现有字段只有 `name / singer / album / time / source`。
3. 现有排序结果通过调整列表成员顺序持久化，不适合直接套到系统生成的只读歌单上。

## 设计方案

### 1. 播放页前景配色

播放页采用“白色前景 + 主题色强调 + 主题色柔和蒙层”的混合方案，而不是把所有元素都直接改成主题色。

原因如下：

1. 主题色本身在浅封面上不一定有足够对比度。
2. 白或近白前景在动态背景场景下更稳，能覆盖封面深浅波动。
3. 主题色更适合承担强调态，而不是整页正文前景。

#### 前景层规则

播放页建立一套专用前景规则：

1. 主文本和主要图标使用高对比前景白，例如接近 `rgba(255,255,255,0.96)`。
2. 次级文本和次级图标使用弱一档的近白，例如接近 `rgba(255,255,255,0.78)`。
3. 时间、次要说明、非激活按钮等不再使用灰色主题 token，而是落到这套近白层级。
4. 激活态、进度、当前歌词高亮等继续用主题色或主题色透明层。

#### 背景层规则

动态背景继续使用模糊封面，但增加统一的播放页蒙层策略：

1. 有封面时，背景基于模糊封面叠加一层偏主题色的柔和蒙层。
2. 这层蒙层主要用于把浅色封面往可读方向拉，而不是把页面整体染成主题色。
3. 对深色封面，只保留较轻压暗，不额外做重主题染色。
4. 对浅色或杂色封面，通过主题色蒙层和轻压暗一起拉出稳定对比。

#### 无封面场景

无封面时不再依赖“普通页面主题背景 + 灰色前景”的组合，而是沿用播放页专用前景策略：

1. 背景仍回退到当前主题背景。
2. 但前景统一使用播放页的白 / 近白规则。
3. 如有必要，可在无封面背景上叠一层轻微暗化，确保白色前景稳定可读。

### 2. 播放页元素映射

以下元素统一切到播放页专用前景规则：

1. 顶部歌曲信息，包括歌名、歌手。
2. 顶部和侧边按钮，包括返回、更多、定时退出之类的图标按钮。
3. 当前时间、总时长、分隔符等播放信息。
4. 控制按钮，包括上一首、下一首、播放暂停。
5. 非激活歌词、歌词时间等目前使用灰色 token 的文本。

以下元素保留主题色强调语义：

1. 当前歌词高亮。
2. 播放进度条与进度点。
3. 已激活或被选中的功能按钮。

### 3. NAS / 媒体来源歌单字体一致性

媒体来源歌单的“只读”属性只影响可执行操作，不影响正文层级。

#### 一致性规则

1. 在“我的列表”左侧列表中，媒体来源歌单名称与普通歌单使用同一层正文色。
2. 在歌曲列表中，歌曲名、序号、副信息、时长等都应回到和普通歌单一致的深色体系。
3. 更多按钮图标也应与普通列表一致，不额外弱化。
4. 普通可见、可播放的媒体来源歌曲，不应呈现出类似“禁用态”的淡灰观感。

#### 异常状态规则

仅以下情况允许弱化：

1. 真正不可播放的歌曲。
2. 菜单中真正不可用的操作项。

即便如此，也只做轻度弱化，不能把整行正文压成接近禁用态。

### 4. 排序能力

#### 入口规则

媒体来源歌单开放排序入口，不再因为歌单是只读就禁用排序。

原因是：

1. 只读歌单不能编辑成员，不代表不能调整显示顺序。
2. 对媒体来源歌单而言，排序是视图层能力，不是数据层编辑。

#### 字段规则

在现有排序字段基础上，新增两个字段：

1. `更新时间`
   默认方向为最新在前。
2. `文件名`
   默认方向为正序。

现有字段保留：

1. 名称
2. 歌手
3. 专辑
4. 时长
5. 来源

#### 数据来源

排序字段的数据来源按以下规则取值：

1. `更新时间` 使用来源文件的修改时间。
2. `文件名` 使用来源文件原始文件名。
3. 对聚合列表或系统生成列表，使用当前展示所依赖的代表来源项来提供这两个字段。

### 5. 排序持久化

每个媒体来源歌单单独记住自己的排序方式。

#### 存储规则

为只读媒体来源歌单新增一个按 `listId` 存储的排序偏好记录，至少包含：

1. `field`
2. `direction`

如果某个歌单没有记录，首次进入时沿用当前默认顺序。

#### 生效时机

以下场景都要重新应用该歌单自己的排序偏好：

1. 重新进入歌单。
2. 手动刷新来源歌单内容。
3. 导入规则同步后歌单内容发生变化。
4. 应用重启后再次打开该歌单。

#### 与普通歌单的边界

普通可编辑歌单保持现有行为：

1. 排序仍然是“真实调整顺序”。
2. 不强行引入“显示排序偏好”的新机制。

媒体来源歌单采用新的视图层排序机制：

1. 只影响显示顺序。
2. 不回写源端。
3. 不改写系统生成列表的成员归属。

## 交互行为

### 媒体来源歌单排序弹窗

排序弹窗延续现有交互结构，只扩展字段和应用逻辑：

1. 用户打开歌单菜单，选择排序。
2. 对媒体来源歌单，弹窗展示可选字段和方向。
3. 用户确认后：
   1. 立即更新当前显示顺序；
   2. 保存该歌单排序偏好；
   3. 后续再次进入继续沿用。

### 默认行为

1. `更新时间` 的默认方向是降序。
2. `文件名` 的默认方向是升序。
3. 其他原有字段继续沿用现有正序 / 倒序 / 随机逻辑。

## 实现边界

### 播放页

优先在播放页已有组件内收敛颜色映射，不扩散为全局主题重构：

1. 页面背景预设仍从 `PageContent` 承接。
2. 播放页内各子组件把灰色前景 token 改为播放页专用前景 token。
3. 如需新增局部辅助常量，可放在播放页组件范围内，不要求扩展整个全局 theme 类型。

### 我的列表 / 媒体来源歌单

只对媒体来源歌单与普通歌单共享的正文层级做校正，不改动已有菜单权限规则：

1. 读写能力判断逻辑保留。
2. 只调整文字、图标、行透明度等视觉表现。

### 排序

新增一个“媒体来源歌单排序偏好”存储层即可，不需要重构整个列表系统：

1. 生成歌单在读取数据后，按 `listId` 查排序偏好并应用。
2. 排序偏好与歌单 `id` 强绑定。
3. 当列表内容刷新时重新套用，而不是把排序结果写回原始列表顺序。

## 验证策略

### 静态 / 单元测试

1. 播放页专用背景预设必须保持“更强模糊 + 更柔和蒙层”。
2. 播放页相关组件不再依赖那批灰色前景 token 作为主要信息层。
3. 媒体来源歌单列表项颜色与普通歌单保持一致。
4. 媒体来源歌单允许打开排序入口。
5. 新增 `更新时间`、`文件名` 两个排序字段。
6. 每个媒体来源歌单排序偏好按 `listId` 持久化。

### 手工验收

1. 有深色封面时，播放页歌曲信息和按钮清晰可读。
2. 有浅色封面时，播放页歌曲信息和按钮仍清晰可读，不被背景吞掉。
3. 无封面时，播放页前景仍保持统一，不退回一套灰色前景。
4. 在“我的列表”里，NAS 生成歌单与普通歌单的字体深浅一致。
5. 打开 NAS 歌单后，歌曲名、序号、副信息和时长不再明显偏灰。
6. 对某个 NAS 歌单选择“按更新时间”，退出再进入后仍保持该排序。
7. 对另一个 NAS 歌单选择“按文件名”，不会影响第一个歌单的排序方式。

## 建议实现顺序

1. 先收敛播放页前景 token 与背景蒙层规则。
2. 再统一媒体来源歌单在“我的列表”和歌曲列表里的字体层级。
3. 最后补媒体来源歌单排序偏好存储与 `更新时间 / 文件名` 两个字段。

这样可以先解决最显眼的可读性问题，再补带状态持久化的排序行为。
